# 02-天气查询页

> 从静态页面进阶到“输入 + 请求 + 展示”的完整小闭环

## 🎯 项目目标

在这个练习中，你需要完成一个简单的天气查询页面。

目标功能：

- 输入城市名称
- 点击按钮后查询天气
- 显示加载中状态
- 成功时显示天气结果
- 失败时显示错误提示

这个练习会比计数器更接近真实项目。

## 🤔 你会练到什么

通过这个练习，你会开始接触这些非常重要的开发能力：

- 输入与状态绑定
- 网络请求思维
- loading / error / data 三态
- 让 AI 帮你处理“不是只画页面”的问题

## 📋 需求说明

请实现一个 SwiftUI 天气查询页，要求：

- 顶部有城市输入框
- 点击“查询天气”按钮后开始请求
- 请求中显示加载状态
- 查询成功后显示城市名、天气描述、温度
- 查询失败后显示错误提示
- 页面结构尽量清晰

如果你暂时不想接真实天气 API，也可以先让 AI 用 mock 数据完成第一版。

## 💬 推荐 Prompt

建议第一轮这样提：

```text
请帮我创建一个 SwiftUI 天气查询页面，要求：
- 顶部有城市输入框
- 点击按钮后查询天气
- 先使用 mock 数据模拟请求
- 要体现 loading、成功、失败 三种状态
- 代码适合初学者理解
- 请解释页面状态是如何管理的
```

这是一个很关键的技巧：**先用 mock 数据跑通页面，再接真实 API。**

## 🛠 推荐开发步骤

### 第一步：只做页面和状态
第一轮先不要着急接真实接口。

目标是先把下面这些做出来：

- 输入框
- 查询按钮
- loading 展示
- 成功结果展示
- 错误提示展示

### 第二步：让 AI 模拟延迟请求
让 AI 用 `Task.sleep` 或简单的异步模拟网络请求。

这样你可以先确认三态逻辑是否正常。

### 第三步：再决定是否接真实 API
等 mock 版本跑通后，再进入下一阶段：

- 增加 WeatherService
- 定义返回模型
- 处理 JSON 解析
- 接真实天气 API

## ✅ 验收清单

请用下面这份清单检查结果：

- [ ] 可以输入城市名称
- [ ] 点击按钮后会进入加载状态
- [ ] 成功时会显示天气数据
- [ ] 失败时会显示错误信息
- [ ] 页面状态切换没有混乱
- [ ] 你能说清楚 loading / error / data 是怎么切换的

## ⚠️ 常见问题

### 问题 1：页面逻辑全写在 View 里，越来越乱
当页面开始复杂时，你可以让 AI 帮你拆出 ViewModel：

```text
请把当前天气查询逻辑拆到一个简单的 ViewModel 中，保持初学者易读。
```

### 问题 2：一接真实 API 就报错
先不要让 AI 重写全部代码，先让它定位问题：

```text
现在 mock 版本没问题，但接真实 API 后失败。
请先分析可能原因：
- 请求地址
- 模型字段
- JSON 解析
- 异步调用位置
然后给我最小修改方案。
```

### 问题 3：状态切换不稳定
常见表现：

- 加载后没有恢复
- 成功和失败同时出现
- 上一次结果没有清空

可以这样要求 AI：

```text
请检查这个页面的状态流转是否清晰：
- 开始请求
- 请求成功
- 请求失败
- 再次请求
并帮我做最小整理。
```

## 🚀 拓展练习

完成基础版后，可以继续增加这些内容：

### 拓展 1：历史搜索
保存最近查询过的城市列表。

### 拓展 2：空状态提示
没有输入城市时给出友好提示。

### 拓展 3：多天气字段
除了温度，再加湿度、风速、体感温度。

### 拓展 4：下拉刷新
重新查询当前城市天气。

### 拓展 5：切换 mock / 真实接口
方便练习不同阶段的开发流程。

## 💡 这个练习最值得学会的事

这个项目的真正重点，不是天气数据本身，而是你开始接触一种更真实的 AI 协作模式：

- 不是一句话生成完整项目
- 而是先做页面骨架
- 再做状态管理
- 再做请求逻辑
- 最后再接真实数据

这种节奏会比“一步到位”稳定很多。

## ✅ 小结

天气查询页是一个非常适合进阶的练习，因为它同时覆盖了：

- 输入
- 状态
- 异步
- 结果展示
- 错误处理

如果你能把这个练习真正做下来，你就已经开始进入“真实开发协作”的节奏了。
