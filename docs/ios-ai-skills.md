# iOS AI Skills 实战清单

> 面向 iOS 开发者的高频 AI 使用场景速查：把 AI 当成结对工程师，而不是一次性代码生成器。

---

## 这页适合谁看

如果你正在做以下工作，这页会非常有用：

- 用 SwiftUI 或 UIKit 开发新页面
- 重构旧项目代码
- 补充 ViewModel / Repository / Network 层
- 排查崩溃、卡顿、状态错乱
- 给现有模块补单元测试

---

## 使用原则

在 iOS 项目中使用 AI，建议遵循以下原则：

1. **给上下文，不给一句话需求**
2. **先要结构，再要实现**
3. **先生成最小可运行版本，再逐轮增强**
4. **让 AI 解释状态流转、线程模型和错误处理**
5. **涉及发布、签名、隐私、支付等内容时必须人工复核**

## 1. SwiftUI 页面生成

### 适用场景
- 列表页
- 详情页
- 表单页
- 设置页
- 空态 / 错误态 / 加载态页面

### Prompt 模板

```text
请使用 SwiftUI 创建一个 [页面名称]，要求：
- 使用 MVVM
- 支持加载态、空态、错误态
- 使用 async/await 获取数据
- 代码按 View / ViewModel / Model 分层
- 给出可预览的 Preview 示例
- 不要使用强制解包
```

### 追问模板

```text
继续优化这个页面：
- 拆分可复用子视图
- 增加骨架屏或 ProgressView
- 增加错误提示和重试按钮
- 说明 @State / @StateObject / @ObservedObject 的选择理由
```

## 2. UIKit 页面迁移到 SwiftUI

### 适用场景
- 老项目技术栈升级
- 局部模块先迁移
- 新页面用 SwiftUI，旧页面仍在 UIKit 中

### Prompt 模板

```text
请将下面的 UIKit 页面迁移为 SwiftUI 实现，要求：
- 保持功能一致
- 保持原有业务流程不变
- 拆分可复用子视图
- 补充 Preview
- 说明状态管理方式
- 如果涉及导航，请使用 NavigationStack 风格
```

### 重点检查
- 生命周期是否一致
- 异步更新是否回到主线程
- 绑定关系是否清晰
- 列表刷新是否稳定

---

## 3. 网络层封装

### 适用场景
- 项目缺少统一请求入口
- 每个页面都在自己拼 URLSession
- 错误处理与解码逻辑重复

### Prompt 模板

```text
请为 iOS 项目设计一个网络层，要求：
- 基于 URLSession
- 支持 async/await
- 支持统一错误处理
- 支持 JSON 解码
- 可扩展请求头、超时、鉴权
- 适合 MVVM 使用
- 给出最小可运行示例
```

### 继续追问

```text
请继续补充：
- APIError 定义
- 通用 Request 协议
- Repository 调用示例
- 单元测试思路
```

---

## 4. ViewModel / MVVM 重构

### 适用场景
- 业务逻辑堆在 View 里
- 页面状态很多，难维护
- 网络、缓存、UI 混在一起

### Prompt 模板

```text
请把下面的 SwiftUI 页面重构为 MVVM，要求：
- View 只保留展示逻辑
- 状态集中到 ViewModel
- 网络调用移到 Repository
- 明确 loading / data / empty / error 四种状态
- 给出重构前后职责说明
```

### 重点收益
- 好测试
- 好复用
- 好讲项目架构

---

## 5. Swift Concurrency 改造

### 适用场景
- 旧代码大量 completion handler
- 回调嵌套深
- 错误处理分散

### Prompt 模板

```text
请把下面基于 completion handler 的代码改为 async/await：
- 保持原有业务逻辑
- 处理错误抛出
- 保证主线程更新 UI
- 补充调用示例
- 说明迁移后的收益
```

### 常见检查点
- 是否遗漏 `@MainActor`
- 是否错误地把耗时工作放在主线程
- 是否保留取消任务的能力

## 6. 崩溃与异常排查

### 适用场景
- 崩溃日志分析
- 数组越界
- 可选值异常
- UI 状态不同步
- 并发访问问题

### Prompt 模板

```text
请分析下面的 iOS 崩溃日志：
- 判断崩溃原因
- 推测可能的业务场景
- 给出修复建议
- 给出防御式编码建议
- 如果可能，请给出修复后的代码示例
```

### 你要提供给 AI 的材料
- 崩溃堆栈
- 触发路径
- 对应代码片段
- 最近改动

---

## 7. 性能优化

### 适用场景
- 启动慢
- 列表滑动卡顿
- 图片内存占用高
- 主线程阻塞

### Prompt 模板

```text
请从 iOS 性能优化角度审查下面的代码，重点检查：
- 主线程阻塞
- 图片加载与缓存
- 列表渲染性能
- 不必要的重复计算
- 状态更新频率过高
并给出可落地优化建议。
```

### 常见优化点
- 大对象不要频繁重建
- 图片异步加载与缓存
- 长列表拆分子视图
- 避免在 body 中做复杂计算

---

## 8. 单元测试补全

### 适用场景
- ViewModel 无测试
- Repository 无 Mock
- 状态切换不稳定

### Prompt 模板

```text
请为下面的 ViewModel 补充 XCTest 单元测试：
- 正常返回
- 空数据返回
- 错误返回
- loading 状态切换验证
- 不依赖真实网络请求
```

### 建议测试对象
- ViewModel
- Repository
- JSON 解码逻辑
- 状态机

## 10. 最佳实践

- 先让 AI 输出目录结构和职责划分
- 再生成最小实现
- 每一轮只改一个问题
- 让 AI 解释“为什么这样设计”
- 关键模块必须自己跑一遍 Build / Test

---

## 小结

对于 iOS 开发者来说，AI 最有价值的地方不是“一键生成整个 App”，而是：

- 帮你快速搭骨架
- 帮你重构成更合理的分层
- 帮你补测试、补错误处理、补边界情况
- 帮你分析崩溃和性能问题

当你把它用成“高质量结对编程助手”时，收益最大。
