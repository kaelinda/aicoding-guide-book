# Codex 使用技巧

> 不只是会打开 Codex，更要学会让它在真正的开发任务里稳定产出

## 🎯 这篇解决什么问题

很多人第一次用 Codex，会感受到它在长任务、代码理解和执行链路上很强，但也会碰到这些问题：

- 不知道该选什么模型
- 不知道推理强度什么时候该调高
- 不知道审批和沙箱该怎么配
- 不知道 `AGENTS.md` 应该写什么
- 不知道怎样把它用成真正的工作流，而不是临时聊天工具

这篇的目标，就是把这些高频问题讲清楚。

---

## 1. 先把 `AGENTS.md` 写好，再让 Codex 长期干活

如果你准备在一个项目里长期使用 Codex，最值得先做的事之一，就是写好项目根目录下的 `AGENTS.md`。

它可以帮助 Codex 更快理解：

- 项目结构
- 代码规范
- 常用命令
- 哪些目录敏感
- 哪些改动必须谨慎
- 哪些输出格式是你团队默认要求

### 建议写进去的内容

优先放这些：

- build / test / lint 命令
- 架构分层说明
- 目录职责
- 命名约定
- 不要碰的区域
- 常见任务的默认流程

比起写很长，更重要的是“稳定、常用、关键”。

---

## 2. 复杂任务优先给“目标 + 约束 + 验收标准”

Codex 很擅长 agentic coding，但如果你只给一句：

```text
帮我优化一下这个模块
```

它就很容易自己补很多假设。

更稳的问法通常是：

```text
目标：把当前 TodoListViewModel 拆得更清晰
约束：
- 不改现有功能行为
- 不新增依赖
- 优先最小修改
验收标准：
- 原有功能仍能运行
- 搜索和删除行为保持一致
- 结构更容易维护
```

### 一个简单原则

> 目标越清楚，约束越明确，Codex 越容易稳定。

---

## 3. 长任务要主动调 reasoning effort

Codex 系列模型支持多档 reasoning effort。

如果你做的是下面这类任务，建议提高推理强度：

- 跨多个文件的重构
- 复杂 bug 排查
- 架构调整
- 长链路任务分解
- 测试策略设计

如果只是：

- 小修一个函数
- 改文案
- 做简单格式整理

就没必要总是开高强度。

### 实用建议

- 日常小任务：低到中等
- 多文件逻辑任务：中到高
- 难 bug / 复杂改造：高

不要所有任务都拉满，不然速度和成本通常都不会太理想。

---

## 4. 审批策略和沙箱模式要配套思考

Codex 的使用体验，很大一部分取决于你怎么设置审批和沙箱。

### 如果你想稳一点

适合：
- 新项目
- 敏感项目
- 第一次做大改动

建议：
- 审批偏谨慎
- 沙箱偏保守
- 先观察 Codex 的操作风格

### 如果你想快一点

适合：
- 熟悉的仓库
- 已经验证过工作流
- 明确知道哪些操作可以自动放行

建议：
- 逐步放宽，不要一次开到很激进
- 先开放局部写权限，再考虑更大范围

### 一个很实用的原则

> 不要只调“快不快”，要一起看“可不可以回退、能不能验证”。

---

## 5. 先让 Codex 规划，再让它执行

尤其是在这些任务里，直接让 Codex 开始改，风险通常会更高：

- 重构
- 批量改名
- 目录调整
- 跨模块联动修改
- 测试体系补全

更稳的方式是分两步：

### 第一步：先规划

```text
请先分析当前问题，并给出一个最小可行方案，不要立即修改代码。
```

### 第二步：再执行

```text
按刚才的方案开始做，每一步修改都尽量可验证。
```

这会明显减少“它一口气改太多”的问题。

---

## 6. 把 Codex 用在它最擅长的任务上

Codex 在这些类型的任务里通常特别强：

### 场景 1：理解现有代码库
让它先解释模块关系、数据流、调用链，而不是上来就改。

### 场景 2：长链路任务拆解
例如一个功能需要跨多个文件实现，先让它做任务分解。

### 场景 3：重构与测试补充
当你已经有一版实现时，Codex 很适合帮你整理结构和补测试。

### 场景 4：PR 级别的代码审查
它很适合先做一轮“问题扫描”，帮你提前发现明显风险。

也就是说，Codex 不只是写代码，更适合做“有上下文、带约束、要持续推进”的任务。

---

## 7. 把高频规则前置写进 `AGENTS.md`

Codex 自动读取 `AGENTS.md`，所以这些东西特别适合前置进去：

- 先解释再改代码
- 优先最小修改
- 不要擅自加依赖
- 大改前先给计划
- 修改后要说明验证方法
- 哪些目录不能动

这样比每次聊天重复说一遍要稳定得多。

---

## 8. 模型选择不要只看“最强”，要看任务形态

OpenAI 目前提供多种面向 coding / agentic coding 的模型，包括 GPT-5.3-Codex、GPT-5.2-Codex、GPT-5-Codex、GPT-5.1-Codex 和更快的小型变体。更高能力的模型通常更适合长链路和复杂推理，小型或更快的模型更适合频繁、小粒度的开发任务。citeturn582739search0turn582739search1turn582739search5turn330613search10

### 实用建议

- 复杂重构 / 难 bug：优先更强模型
- 高频小改 / 快速迭代：优先更快模型
- 长任务拆解：优先推理更强的模型

关键不是“永远选最强”，而是让模型能力和任务复杂度匹配。

---

## 9. 让 Codex 产出“检查清单”，不只是代码

很多人只让 Codex 写代码，但更高价值的用法是让它顺手产出：

- 验收清单
- 回归检查点
- 风险说明
- 测试策略
- PR 总结

例如：

```text
修改完成后，请额外输出：
1. 验收清单
2. 可能的回归风险
3. 推荐补的测试
```

这会把一次编码任务升级成一次更完整的工程协作。

---

## 10. 适合长期使用的 Codex 组合技

### 组合 A：`AGENTS.md` + 约束式 Prompt
解决“上下文不稳定”和“它自己脑补”的问题。

### 组合 B：规划 → 执行 → 验收
解决复杂任务一口气做太大的问题。

### 组合 C：推理强度按任务调整
解决所有任务都用同一种配置的低效问题。

### 组合 D：审查输出 + 测试补全
解决“代码写完了，但不够稳”的问题。

---

## ⚠️ 最容易踩的坑

### 坑 1：把 Codex 当成只会写代码的工具
它在理解代码库、做计划、做 review 上也很强。

### 坑 2：没有 `AGENTS.md`
没有项目规则前置，长期协作会越来越费劲。

### 坑 3：复杂任务不做分阶段
一口气让它做大事，最容易造成失控 diff。

### 坑 4：审批和沙箱开得太随意
速度提升了，但风险也一起放大。

---

## ✅ 小结

Codex 最值得掌握的，不只是“写代码能力”，而是这套长期协作方式：

- 用 `AGENTS.md` 固定项目规则
- 用清晰的目标 / 约束 / 验收标准发任务
- 用合适的推理强度匹配任务复杂度
- 用规划 → 执行 → 验收的节奏推进复杂工作
- 让它同时产出代码、检查清单和风险说明

记住这句话：

> **Codex 的强项，不只是生成代码，而是持续推进复杂开发任务。**
