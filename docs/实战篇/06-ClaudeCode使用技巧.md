# Claude Code 使用技巧

> 不只是会用 Claude Code，更要学会把它用稳、用顺手

## 🎯 这篇解决什么问题

很多人第一次用 Claude Code，会有两种常见感觉：

- 很强，但不稳定
- 能做很多事，但总觉得“有点不受控”

这通常不是 Claude Code 不好用，而是还没有建立一套更成熟的使用方法。

这篇不讲安装，而是专讲 **如何把 Claude Code 用得更稳、更省上下文、更适合长期协作**。

---

## 1. 先用 `/init` 建立项目级记忆

如果你准备长期在一个项目里使用 Claude Code，第一件事不是马上让它写功能，而是先运行：

```bash
/init
```

这样可以帮助你建立项目级的 `CLAUDE.md`，把这些高频信息固定下来：

- 项目结构
- 常用命令（build / test / lint）
- 编码规范
- 架构约定
- 不要碰的目录或文件

这会明显减少你每次重复解释上下文的成本。

### 建议写进 `CLAUDE.md` 的内容

优先放这些：

- 如何构建和测试
- 目录职责
- 命名习惯
- 哪些模块比较脆弱
- 哪些改动必须谨慎

不要把它写成百科全书，重点是“高频、稳定、关键”。

---

## 2. 学会区分“项目记忆”和“个人记忆”

Claude Code 支持多层记忆，包括项目级 `./CLAUDE.md` 和用户级 `~/.claude/CLAUDE.md`。

最实用的做法是：

### 项目记忆放什么

放团队共享、项目相关的东西：

- 架构说明
- 代码规范
- 常用命令
- 测试方式
- 危险区域说明

### 个人记忆放什么

放你自己的偏好：

- 你喜欢的注释风格
- 你常用的排查方式
- 你喜欢 Claude 输出的格式
- 你个人的工作习惯

这样不会把个人偏好污染到团队项目。

---

## 3. 高频命令优先用 Slash Commands 管理

Claude Code 内置了不少非常实用的 slash commands，例如：

- `/memory`
- `/model`
- `/permissions`
- `/review`
- `/mcp`
- `/compact`
- `/agents`

如果你某些 Prompt 经常重复使用，建议做成 **自定义 slash command**，而不是每次重新打一遍。

### 适合做成命令的内容

例如：

- Bug 排查
- 提交前 review
- 重构前检查
- 测试补全
- PR 总结

这会显著减少重复劳动。

---

## 4. 上下文长了以后，优先主动 `/compact`

Claude Code 会话一长，最常见的问题不是“不会做”，而是：

- 背景越来越多
- 旧上下文干扰新任务
- 它开始抓不住当前重点

这时候不要硬顶着继续聊，建议主动执行：

```bash
/compact
```

或者给一点聚焦提示，例如：

```bash
/compact 只保留当前 bug 排查相关上下文
```

这样通常会比继续在长会话里硬做更稳。

---

## 5. 把重复任务交给 Subagents

Claude Code 支持自定义 subagents，它们有独立上下文窗口，可以做任务级分工。

这特别适合下面这些场景：

- 一个 agent 专门查 bug
- 一个 agent 专门做 review
- 一个 agent 专门做重构规划
- 一个 agent 专门写测试

### 推荐原则

不要做一个“全能代理”。

更稳的方式是：

- 一个 subagent 只负责一类任务
- 给它非常清晰的 description
- 只给必要工具权限

越单一、越具体，效果通常越好。

---

## 6. Hooks 适合做“强约束”，不要只靠嘴说

如果你有一些规则是“每次都必须执行”的，例如：

- 某些命令执行前必须确认
- 修改文件后自动跑格式化或 lint
- 会话启动时自动加载某些信息

这类场景适合用 Claude Code hooks，而不是每次靠 prompt 提醒。

### 适合上 hooks 的场景

- 阻止危险命令
- 自动记录命令日志
- 自动加载项目上下文
- 在某些工具调用前做约束检查

### 但要注意

Hooks 会带来自动执行能力，所以一定要审查自己的 hooks 脚本，别把风险自动化了。

---

## 7. Claude Code 最稳的节奏：先计划，再执行

如果任务稍微复杂，建议不要一上来就说“直接改”。

更稳的方式是：

### 第一步：先让它分析

```text
请先分析当前问题，不要立即修改代码。
```

### 第二步：再让它给计划

```text
请给出一个最小可行方案，并说明风险点。
```

### 第三步：最后再允许它动手

```text
按这个方案做最小修改。
```

这套节奏尤其适合：

- 重构
- 批量改动
- 状态管理问题
- 跨多个文件的问题

---

## 8. Review 时不要只让它“夸”，要给明确审查维度

很多人会直接说：

```text
review 一下
```

更有效的是限定 review 维度，例如：

```text
请从下面 4 个角度 review：
1. 逻辑风险
2. 边界情况
3. 状态流是否清晰
4. 是否存在过度设计
```

这样 Claude Code 的 review 会实用很多。

---

## 9. 大改前先缩小范围

Claude Code 非常擅长大范围操作，但真正稳定的使用方式往往是：

- 先改一个文件
- 再改一个模块
- 再扩大到全项目

而不是：

- 一次性大重构
- 一次性跨很多文件
- 一次性改命名、逻辑、结构、测试

### 一个简单原则

> 能局部验证的改动，优先不要全局做。

---

## 10. 适合长期使用的 Claude Code 组合技

如果你想把 Claude Code 真正变成日常工具，最推荐的组合是：

### 组合 A：`/init` + `CLAUDE.md`
解决重复解释上下文的问题。

### 组合 B：`/compact` + 小步任务
解决长会话漂移的问题。

### 组合 C：Subagents + 明确分工
解决复杂任务互相污染上下文的问题。

### 组合 D：Hooks + 安全约束
解决“规则每次都要重复说”的问题。

---

## ⚠️ 最容易踩的坑

### 坑 1：什么都想一次做完
Claude Code 很强，但一次塞太多目标，仍然容易跑偏。

### 坑 2：没有项目记忆
没有 `CLAUDE.md` 的长期项目，协作成本会越来越高。

### 坑 3：会话太长却不压缩
上下文不整理，后面通常会越来越不稳。

### 坑 4：把自动化能力开太大
尤其是 hooks 和大范围命令执行，越自动越要谨慎。

---

## ✅ 小结

Claude Code 最值得掌握的，不是某个单一命令，而是这套工作方式：

- 用 `CLAUDE.md` 固定上下文
- 用 slash commands 固定高频流程
- 用 `/compact` 管理会话长度
- 用 subagents 分工
- 用 hooks 固化规则
- 用“小步可验证”的方式推进复杂任务

记住这句话：

> **Claude Code 用得越久，越要把“对话”升级成“工作流”。**
