# 13-Claude Code 我的常用 Prompt

> 日常高频使用的 7 条 Prompt 模板。每条给出：**什么时候用 / 直接可粘 / 期望输出 / 常见偏差 / 如何纠偏**。

---

## 为什么要固定 Prompt 模板

临时想 Prompt 有两个问题：

1. **每次都在重新发明轮子**：同样的请求换一种说法，Claude 的表现可能差很多
2. **没有控制变量**：Prompt 自由度越高，Claude 越倾向"发挥"

把固定场景沉淀成模板后，你就只需要调"今天这个特殊情况加什么"，而不是从零想"怎么说"。

建议配合 Claude Code 的 **`/command`** 或 **Skill** 机制把下面这些 Prompt 固化（参见 [05-ClaudeCode常用Skills.md](05-ClaudeCode常用Skills.md)）。

---

## 1. 项目理解 Prompt

**什么时候用**：进入一个陌生项目 / 回到搁置 2 周以上的项目 / PR review 前。

```text
先不要改代码，分析这个项目：
1. 技术栈和核心依赖
2. 目录结构和主要模块职责
3. 数据流（用户输入 → UI → 状态 → API → 持久化）
4. 最近 30 个 commit 在做什么主题
5. 列出 3 个"如果要改 X 功能，你要先看哪几个文件"的典型任务路径

输出用 markdown 表格，不要长篇叙述。
```

**期望输出**：一份 2-3 分钟能读完的项目地图。

**常见偏差**：

- Claude 倾向写"业务价值""架构优雅度"这种虚的 → **明确要求"只讲事实、不讲好坏"**
- 技术栈列得太浅（只列 Swift / SwiftUI）→ **追问具体版本、具体并发模型、具体测试框架**

---

## 2. Bug 排查 Prompt

**什么时候用**：看到一个不明显的 bug，想先搞清楚方向再动手。

```text
Bug：<一句话描述现象>
复现：<步骤>
预期 vs 实际：<差异>

先不要改代码：
- 判断问题类型（状态 / 并发 / 生命周期 / 网络 / 依赖 / 环境）
- 列出你要读的文件（最多 5 个，按优先级）
- 给 3 个可能的根因，按可能性排序
- 每个根因给一个"最快速验证它是不是"的方法
```

**期望输出**：3 条候选根因 + 3 个验证动作。

**常见偏差**：

- 直接给 patch 让你试 → **重申"先不要改代码"**，猜测性修复是 bug 再生的元凶
- 只说"检查一下 xxx" 不给具体验证方法 → **追问"具体跑什么命令 / 点什么按钮可以验证这条"**

---

## 3. 最小修改 Prompt

**什么时候用**：已经定位到根因，要做修复 patch 时。

```text
根因：<一句话>
要求：只做最小必要修改：
- 不改无关代码
- 不"顺手"重构
- 不改命名 / 格式 / 缩进
- 改完附：改了哪些文件、每处几行、为什么改

如果你认为"顺手改 xxx 会更好"，独立列出来作为"可选优化"，不要直接改。
```

**期望输出**：一个聚焦的 diff + 一份可选优化清单（由你决定要不要做）。

**常见偏差**：

- 把 format 改动混进来 → **必要时直接说"保持原文件的所有换行和缩进不变"**
- 名义上最小，实际上重写了整个函数 → **看到 diff 异常大时直接拒绝并让它重做**

---

## 4. 方案设计 Prompt

**什么时候用**：需求比较大（> 100 行改动量），不想一头扎进去。

```text
需求：<贴完整需求>
约束：<已知约束，如"必须兼容 iOS 15""不能引入新依赖">

先不要写代码。给两个方案：
- 方案 A：保守（在现有代码上最小改动）
- 方案 B：彻底（重新设计相关模块）

每个方案说明：
- 改动范围（文件数、新增 / 修改 / 删除）
- 实现步骤（3-5 步）
- 风险点和对应 mitigation
- 回滚难度

最后给你的推荐和理由（一句话）。
```

**期望输出**：两个对比方案 + 推荐。

**常见偏差**：

- 方案 A 和 B 差别不大 → **要求方案 B 必须"重构范围显著大于 A"，否则只给一个**
- 推荐一边倒 → **让它至少写 2 条"虽然不推荐，但选 X 方案合理的场景"**

---

## 5. 拆任务 Prompt

**什么时候用**：方案选定后，把它落成 Builder 可执行的清单。

```text
基于下面的方案，拆成 3-5 个可执行步骤。
方案：<贴 #4 的输出>

每步给：
- 产出文件（绝对路径）
- 改动量（预估行数）
- 验收标准（怎么跑、看什么结果算过）
- 依赖关系（这步依赖哪几步）

用 markdown 表格。
```

**期望输出**：一张任务表，每行代表一个可独立提交的 commit。

**常见偏差**：

- 拆出来 10+ 步 → **合并到 3-5 步，或按 commit 粒度重拆**
- 验收标准模糊（"代码能跑"）→ **要求具体到"跑 xxx 命令，看到 yyy 输出"**

---

## 6. 验证 Prompt

**什么时候用**：一个 step 写完之后、声称"done"之前。

```text
刚才的改动：
- 改了哪些文件：<列出>
- 改动摘要：<一句话>

现在请：
1. 给出具体验证命令（xcodebuild / swift test / 手工点击路径）
2. 给出 3 个边界测试用例（不是 happy path）
3. 说明有哪些场景这个改动"看起来没问题但可能会出问题"

不要声称"应该没问题"。只写能被验证的东西。
```

**期望输出**：可执行的验证清单 + 风险声明。

**常见偏差**：

- 输出里出现"应该能"、"理论上"、"可能需要" → **把这些词当作 red flag，要求它把"应该"换成"我验证过 / 我没验证"**

---

## 7. 总结 Prompt

**什么时候用**：一轮任务收尾、准备开下一个会话前。

```text
这一轮任务：<目标>

请总结：
1. 改了什么（文件 + 核心改动）
2. 根因是什么（如果是 bug）/ 设计要点（如果是新功能）
3. 验证过什么、没验证什么
4. 遗留的风险或 TODO
5. 值得写进 CLAUDE.md 的经验（不超过 3 条）

输出用 markdown，可以直接粘到 commit message 或 PR description。
```

**期望输出**：一段可复用的总结。

**常见偏差**：

- 总结变成了工作汇报（"本次任务圆满完成" 这类） → **明确"只写事实，不写评价"**
- 遗留风险写成"无" → **追问"假设用户以 xxx 方式操作会发生什么"**

---

## 配合使用的节奏

单独用一条 Prompt 威力一般，真正有用的是**把它们串成一天的工作流**：

```text
早：#1 项目理解（如果是回来的项目）
午前：#2 bug 排查 → #4 方案设计 → #5 拆任务
午后：#3 最小修改（每 step 一次）→ #6 验证（每 step 一次）
傍晚：#7 总结（一天一次，进 commit / 进 CLAUDE.md）
```

---

## 怎么把这些 Prompt 沉淀下来

有几个选择，按投入从低到高：

1. **存在 `snippets.md`**：开 session 时复制粘贴，零成本，但每次要手动选
2. **做成 Claude Code `/command`**：`.claude/commands/` 下建文件，敲 `/bug` 直接带出 Prompt #2
3. **做成 Skill**：进一步让 Claude 自动在合适场景召唤这些 Prompt（见 `05-常用 Skills`）
4. **团队版**：把团队通用的几条放进 `CLAUDE.md`（项目级）或 `~/.claude/CLAUDE.md`（个人级）

我自己的选择是 2 + 4：高频 7 条做成 `/command`，团队约定写进项目 `CLAUDE.md`。

---

## 一句话总结

**把常用操作沉淀为固定 Prompt，不是为了少打字，是为了减少变量。**

同一件事每次用同一个 Prompt，Claude 的表现才可预测 —— 可预测是工程化的前提。
