# 06-Claude Code 上下文管理

> 学会管理上下文，Claude Code 才会从“偶尔答得好”变成“持续稳定地完成任务”。

---

## 为什么上下文管理这么重要

很多人觉得 Claude Code 不稳定，常见原因并不是模型能力不够，而是上下文被污染了：

- 任务做着做着跑偏
- 已经确认的约束被遗忘
- 前面说的是修 bug，后面变成重构
- 一轮里塞了太多目标
- 对话越来越长，但有效信息越来越少

Claude Code 的优势在于它可以持续读代码、执行命令、跟踪任务；
但前提是：**你要让上下文保持清晰、收敛、可控。**

---

## 1. 把上下文分成 4 层

推荐你把上下文拆成这 4 层来管理：

### 第一层：任务目标
当前这轮到底要完成什么？

例如：
- 修复登录页重复请求
- 给用户模块补测试
- 把聊天页重构成 MVVM

### 第二层：项目背景
这个项目是什么技术栈、什么目录结构、有什么约束。

例如：
- SwiftUI + MVVM
- 不允许强制解包
- 不引入新依赖

### 第三层：局部上下文
本轮真正相关的文件、模块、命令。

例如：
- `LoginView.swift`
- `LoginViewModel.swift`
- `AuthRepository.swift`

### 第四层：运行结果
本轮执行过什么命令、出现什么错误、哪些假设已被验证。

例如：
- 跑过测试
- 构建失败点
- 复现步骤是否成立

---

## 2. 一轮只做一件事

Claude Code 很适合连续推进，但不适合同一轮里混杂多个目标。

### 不推荐

```text
修一下登录 bug，顺便把用户模块重构掉，再补一下测试，最后更新文档。
```

### 推荐

```text
这一轮只做一件事：修复登录页重复请求。
先定位根因，给出最小修复方案，改完后跑相关验证。
不要做重构，不要顺手改其他模块。
```

当你把任务切小，Claude 的稳定性会明显提升。

---

## 3. 每一轮都要重申任务边界

Claude Code 是持续对话式工具，所以边界要不断提醒。

推荐在每轮开头补一句：

```text
当前任务边界：
- 只处理登录页重复请求
- 不改动公共网络层
- 不引入新依赖
- 不重构无关代码
```

这样可以降低“顺手大改”的风险。

---

## 4. 把“已确认事实”单独列出来

当任务稍复杂时，建议你明确告诉 Claude：哪些信息已经确认，不要反复推翻。

```text
已确认事实：
1. 问题只在 iOS 端出现
2. 接口本身没有重复返回
3. 重复请求发生在页面进入时
4. 当前项目使用 SwiftUI + MVVM
```

这一步非常有用，因为它能把对话从“猜测”拉回“基于证据推进”。

---

## 5. 让 Claude 先总结再继续

如果一轮对话已经比较长，不要急着继续追加需求。
先让它做一次压缩总结。

```text
请先总结当前上下文：
- 当前任务是什么
- 已完成什么
- 尚未完成什么
- 哪些风险还存在
然后再继续下一步。
```

这相当于人为做一次“上下文整理”。

---

## 6. 长任务要切阶段

### 推荐阶段化方式

#### 阶段 1：理解
- 看目录
- 看架构
- 找入口
- 明确问题范围

#### 阶段 2：分析
- 找根因
- 列方案
- 评估风险

#### 阶段 3：执行
- 做最小修改
- 跑构建 / 测试
- 输出结果

#### 阶段 4：收尾
- 总结改动
- 提出后续建议
- 记录回归项

---

## 7. 上下文太长时，要主动“压缩”

对话轮数一多，容易出现：
- Claude 记得细节但忘了目标
- 或记得目标但细节混乱

这时候你可以直接说：

```text
请压缩上下文，只保留以下信息：
- 当前任务目标
- 已确认事实
- 当前相关文件
- 已尝试过的方法
- 下一步建议
```

这样能把对话重新收敛到高价值信息上。

---

## 8. 不要把“历史所有细节”都留在上下文里

很多人误以为：
> 上下文越多越好。

其实不是。

无关历史会造成：
- 干扰当前判断
- 增加误改概率
- 让 Claude 更容易误以为你要继续做旧任务

所以建议：
- 当前轮只保留当前问题需要的信息
- 旧任务完成后显式切换
- 新任务重新定义边界

---

## 9. 新任务开始时，要明确“上下文切换”

推荐用这种句式：

```text
上一个任务已经结束。
现在开始新任务：为聊天模块补测试。
请不要继续沿用上一个 bug 修复任务的假设。
```

这一步能显著减少串任务问题。

---

## 10. 善用文件作为稳定上下文载体

如果某些信息非常关键，不要只放在对话里。

更稳妥的做法是写进文件：
- `AGENTS.md`
- `README.md`
- `docs/architecture.md`
- 模块自己的说明文档

这样 Claude Code 每次进入项目都能重新读取，而不是完全依赖聊天历史。

---

## 11. 推荐的上下文模板

你可以直接复制这段当作每轮任务开头：

```text
当前任务：
[一句话说明目标]

任务边界：
- [边界 1]
- [边界 2]
- [边界 3]

已确认事实：
1. [事实 1]
2. [事实 2]
3. [事实 3]

相关文件：
- [文件 1]
- [文件 2]

执行要求：
1. 先分析
2. 再给方案
3. 然后再改代码
4. 最后运行验证并总结
```

---

## 12. Claude Code 上下文管理常见错误

### 错误 1：任务目标混杂
修 bug + 重构 + 写文档一起做。

### 错误 2：没有边界
没说清楚哪些文件能改，哪些不能改。

### 错误 3：不做总结
一轮结束后直接跳下一轮，导致上下文越来越乱。

### 错误 4：旧任务没关掉
上个任务其实结束了，但 Claude 还在沿用旧假设。

### 错误 5：把所有历史都留着
结果噪音越来越大。

---

## 13. 一套很实用的工作流

### 修 bug
1. 先声明当前任务
2. 列已确认事实
3. 指定相关文件
4. 让 Claude 先分析根因
5. 只做最小修复
6. 验证后做总结

### 新功能
1. 先看目录和现有模式
2. 列实现计划
3. 先做最小版本
4. 再补边界和测试

### 重构
1. 先给两个方案
2. 选保守或彻底方案
3. 分轮执行
4. 每轮都验证行为不变

---

## 小结

上下文管理的本质，不是“塞更多信息”，而是：

- 明确目标
- 控制边界
- 只保留高价值事实
- 在关键节点主动压缩总结
- 用文件而不是纯对话承载长期约束

当你这样使用 Claude Code，它会稳定很多，尤其适合复杂工程任务。
