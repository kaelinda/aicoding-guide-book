# 04-Claude Code 使用技巧

> 从“会用 Claude Code”进阶到“高效用 Claude Code”。

---

## 为什么很多人用了 Claude Code 还是不稳定

不是工具不行，而是常见使用方式有问题：

- 一上来就让它“大改整个项目”
- 没给清楚上下文
- 没有让它先做计划
- 不验证结果，直接继续下一轮
- 把 Claude Code 当成一次性代码生成器

更好的思路是：

> 把 Claude Code 当成一个能看代码、会执行命令、会逐步推进任务的工程搭档。

---

## 1. 先让它“理解项目”，再让它“改项目”

很多失败都发生在第一步：Claude 还没理解目录、技术栈、约束，就直接开始写代码。

推荐起手式：

```text
先不要改代码。
请先完成下面几件事：
1. 阅读项目目录结构
2. 判断技术栈和架构
3. 找出主要入口文件
4. 总结已有代码风格
5. 给出你对这个项目的理解
```

适用场景：
- 新接手项目
- 老仓库很久没看
- 团队项目目录复杂

---

## 2. 先做计划，再执行

不要直接说“帮我重构”。
先让它拆步骤。

推荐写法：

```text
请先不要动代码。
把这个任务拆成 5 到 8 个步骤，说明：
- 每一步要改哪些文件
- 风险点是什么
- 应该先验证什么
等我确认后再开始执行。
```

好处：
- 避免一次改太多
- 便于你中途纠偏
- Claude 自己也更稳定

---

## 3. 大任务要分轮推进

### 错误示范

```text
把整个项目重构成 MVVM，并补齐测试和文档。
```

### 更好的写法

```text
第一轮：先只分析用户模块，识别当前分层问题。
第二轮：把用户模块重构成 MVVM。
第三轮：只补这个模块的测试。
第四轮：总结这次改动对整个项目的启发。
```

Claude Code 很适合**连续推进**，不适合一口吞掉一个巨大任务。

---

## 4. 让它先读，再改，再测

推荐固定节奏：

1. 读代码
2. 说理解
3. 给计划
4. 动手改
5. 跑测试 / 构建
6. 总结改动

你可以直接这样说：

```text
请按下面流程执行：
1. 先阅读相关文件
2. 简述问题根因
3. 提出最小修改方案
4. 修改代码
5. 运行相关测试或构建命令
6. 汇总改动与风险
```

这比单纯说“修这个 bug”稳定得多。

---

## 5. 用“最小修改”约束它

Claude Code 有时会为了“顺手优化”改太多。
所以你要明确范围。

```text
请只做最小必要修改：
- 不要改动无关文件
- 不要顺手重命名一堆变量
- 不要引入新依赖
- 不要调整现有目录结构
```

适合：
- 修 bug
- 发版前修复
- 热修复
- 老项目保守改动

---

## 6. 明确“不要做什么”

很多高质量协作，不是因为你说得多，而是你把边界说清楚了。

推荐补这一段：

```text
约束：
- 不要修改公开 API
- 不要改变现有业务行为
- 不要删除测试
- 不要新增第三方依赖
- 如果需要大改，请先停下来说明原因
```

这尤其适合：
- SDK
- 公共组件库
- 团队协作仓库

---

## 7. 遇到 bug 时，让它先判断根因类型

不要直接让它修。
先让它分类。

```text
请先不要改代码。
先判断这个问题更像是：
- 状态管理问题
- 并发问题
- 生命周期问题
- 数据边界问题
- 网络问题
并说明你为什么这么判断。
```

这样能有效避免“瞎修”。

---

## 8. 重构任务一定要让它保留行为不变

推荐模板：

```text
请重构这段代码，但遵守以下要求：
- 保持现有功能不变
- 保持外部调用方式不变
- 优先改善可读性和分层
- 如果需要改变接口，先说明原因，不要直接改
```

---

## 9. 让它输出“改动摘要”

每轮结束后，让 Claude Code 输出总结，非常有价值。

```text
请总结这轮改动：
- 改了哪些文件
- 每个文件改动目的是什么
- 运行了哪些命令
- 还有什么风险未解决
```

这能帮你：
- 快速 review
- 写 commit message
- 写 PR 描述
- 记录变更历史

---

## 10. 善用 AGENTS.md

Claude Code 很适合配合 `AGENTS.md` 使用。

建议在里面放：
- 技术栈
- 目录约定
- 编码规范
- 常用命令
- 明确禁止事项

例如：

```markdown
## 技术栈
- SwiftUI
- MVVM
- URLSession

## 编码规范
- 不使用强制解包
- 业务逻辑不放在 View 中
- ViewModel 负责状态管理

## 常用命令
- 构建：xcodebuild -scheme App
- 测试：xcodebuild test -scheme App
```

---

## 11. 善用 CLAUDE.md（Claude Code 专属）

`AGENTS.md` 是跨工具通用约定，而 `CLAUDE.md` 是 Claude Code **原生识别的项目指令文件**。每次开新会话，Claude Code 会自动把它读进上下文。用好它，比你反复在对话里交代背景省很多 token。

### 三种层级，从近到远覆盖

Claude Code 会按顺序合并下面三份文件（越靠前的优先级越高）：

| 文件 | 位置 | 用途 | 是否入 git |
|------|------|------|----------|
| `CLAUDE.local.md` | 项目根 | 你个人的临时指令、私有 scratch | ❌ 加进 `.gitignore` |
| `CLAUDE.md` | 项目根 | 团队共识的项目指令 | ✅ 入库 |
| `~/.claude/CLAUDE.md` | 用户家目录 | 你跨项目的通用偏好 | ✅ 个人 dotfiles |

### 项目 `CLAUDE.md` 里该放什么

和 `AGENTS.md` 互补，重点放 Claude Code **一上来就需要知道**的信息：

- **项目一句话定位**：是什么、给谁用、关键技术栈
- **构建 / 测试 / 运行命令**：`npm run dev`、`xcodebuild test ...`，避免它现查现试
- **架构大图**：哪个目录对应哪个职责、数据从哪流到哪（2-3 段就够）
- **硬约束**：不许用某个库、不许改某个目录、commit 规范、分支命名规则
- **"不要做什么"清单**：比"要做什么"更有价值

### 实用小技巧

- **用 `@path` 引入外部文件**，保持主文件简洁：
  ```markdown
  项目编码规范见 @docs/coding-style.md
  ```
  Claude Code 会把 `@` 引用的文件内容一起加载。

- **控制在 ~200 行以内**。`CLAUDE.md` 每次会话都占 context 预算，写成小说反而挤掉代码空间。

- **不要塞 README**。README 给人看，`CLAUDE.md` 给 AI 看；让它 `@README.md` 引用就够了，别复制。

- **绝不写密钥 / token**。文件会被 AI 完整读入，也可能被日志/分享泄露。

- **发现 AI 反复踩同一个坑 → 补一条进 `CLAUDE.md`**。这份文件应该随项目长出来，而不是一次写死。

### 常见错误

- ❌ 把整个架构文档拷进去 → 吃光 context
- ❌ 把"最佳实践"那种通用废话堆一堆 → AI 本来就懂，浪费 token
- ❌ 项目一变 CLAUDE.md 不跟着改 → 过时指令比没指令更糟

> 💡 想系统学写，见 [进阶篇 · Claude Code 写法教程](../../进阶篇/09-ClaudeCode写法教程/claude.md-教程.md)。

## 12. 让它做“代码审查”而不是只写代码

Claude Code 的另一个高价值用途，是 review。

你可以这样问：

```text
请审查这个模块，重点关注：
- 是否有状态管理问题
- 是否有重复请求风险
- 是否有强制解包
- 是否有主线程阻塞
- 是否有明显可测试性问题
```

适合：
- 提交前自检
- 新人代码 review
- 老模块体检

---

## 13. 让它先给“两个方案”，再选一个

面对稍复杂的改动，不要一上来只要一个答案。

```text
请给出两种可行方案：
- 保守方案
- 更彻底的方案
并比较它们的风险、工作量和适用场景。
```

这会大幅提升你对任务的掌控感。

---

## 14. Claude Code 适合做什么

特别适合：
- 分析陌生代码库
- 修局部 bug
- 做渐进式重构
- 批量改小问题
- 补测试
- 执行命令并验证结果
- 输出改动总结

不适合直接放任：
- 一口气重构整个仓库
- 不设边界地改架构
- 涉及支付、隐私、上线配置时无人复核

---

## 15. 推荐工作流

### 新功能开发
1. 先让 Claude 分析目录
2. 要它给实现计划
3. 先实现最小版本
4. 跑构建 / 测试
5. 再补边界和文档

### 修 bug
1. 先让 Claude 判断问题类型
2. 缩小相关文件范围
3. 做最小修复
4. 跑测试或复现步骤
5. 输出根因和回归建议

### 代码审查
1. 指定模块
2. 指定检查维度
3. 要求给风险级别
4. 最后输出可执行建议清单

---

## 16. 一组很好用的短句

你可以直接复制这些给 Claude Code：

```text
先别改代码，先分析。
```

```text
请给我最小改动方案。
```

```text
请先列步骤，再开始执行。
```

```text
只改和这个问题直接相关的文件。
```

```text
改完以后运行相关测试，并总结风险。
```

```text
如果你发现需要大改，请先停下来说明原因。
```

---

## 小结

Claude Code 用得好的关键，不是“prompt 花哨”，而是这几点：

- 先分析，再执行
- 先计划，再修改
- 控制范围，限制副作用
- 每轮都验证
- 让它输出总结

当你这样使用它时，Claude Code 会非常像一个稳定的工程搭档，而不是一个偶尔灵光一现的代码生成器。
