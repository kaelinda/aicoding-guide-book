# 12-Claude Code 个人工作流

> 我日常用 Claude Code 的三个标准流程：修 bug、做新功能、做重构。**共用一条主线**。

---

## 共用主线

不管什么任务，都是这五个动作：

```text
先分析 → 再计划 → 小步执行 → 每步验证 → 最后总结
```

看起来像废话，但 90% 的"AI 失控"都是因为跳了其中某一步：

| 跳过的环节 | 典型故障 |
|------|------|
| 跳过"先分析" | 改完才发现根因在别的文件 |
| 跳过"再计划" | 一轮对话改 5 个文件，回滚困难 |
| 跳过"小步执行" | 单次改动太大，review 成本高 |
| 跳过"每步验证" | 最后跑起来全是编译错误 |
| 跳过"最后总结" | 下次遇到相同问题还会再踩 |

下面三个流程都是这条主线的具体化。

---

## 流程 A：修 bug

### Step 1 — 定义问题（不要急着贴代码）

```text
问题描述：<一句话说现象>
复现步骤：
1. ...
2. ...
预期：...
实际：...

先不要改代码：
- 判断问题类型（状态 / 并发 / 生命周期 / 网络 / 依赖版本）
- 列出需要读的文件
- 给 2-3 个可能的根因，按可能性排序
```

**为什么不直接贴代码让它修**：让 Claude 先列文件，你能判断它是否看到了该看的地方。
如果它没提到关键文件，说明信息不够，这时候补信息比让它瞎猜便宜。

### Step 2 — 判断类型

iOS 常见 bug 大致 4 类，对症下药：

- **状态问题**：@State / @StateObject / @Published 用错 → 看 View 和 ViewModel
- **并发问题**：多个 Task 改同一状态 → 看 actor / MainActor / async 边界
- **生命周期**：onAppear / init 被意外重入 → 看父 View 的 id / NavigationPath
- **网络 / 数据**：缓存 / 重试 / 幂等性 → 看 request 发出的位置

### Step 3 — 最小修复

```text
只做最小必要修改：
- 不改无关代码
- 不顺手优化
- 改完附验证步骤
```

关键词是"最小"。Claude 的默认倾向是"既然在看这个文件，把代码风格也统一一下吧" —— 不要。

### Step 4 — 验证

```bash
# 编译 + 单测
xcodebuild test -scheme MyApp -destination 'platform=iOS Simulator,name=iPhone 15'
# 或最低限度
xcodebuild build -scheme MyApp -destination 'platform=iOS Simulator,name=iPhone 15'
```

单测覆盖不到的改动（UI / 生命周期），**必须手工复现一次**。
不信任"逻辑上应该对"，只信任"我刚才真跑过一次"。

### 该停下的信号

- Claude 改了 3 次还修不好 → 停，换思路（可能根因判断错了，回 Step 1）
- diff 超过 50 行 → 停，问它能不能拆小
- 它开始说"可能是 xxx 导致的，让我修改一下试试" → 停，**猜测性修复是 bug 再生的主要原因**

## 流程 B：做新功能

### Step 1 — 看现有实现

```text
先不要写代码：
- 列出项目里类似功能的实现位置（比如已有的 FooView / BarViewModel）
- 说明那套约定是什么（命名 / 分层 / 状态管理 / 测试位置）
- 我要加的新功能应该怎么对齐这套约定
```

iOS 项目经常并存 MVC / MVVM / TCA / @Observable，不对齐现状就会长出四不像。

### Step 2 — 给计划

```text
把新功能拆成 3-5 步，每步给：
- 产出文件
- 验收标准
- 预估改动量
```

3-5 步是经验值。

- **少于 3 步**：可能功能太简单，直接写；或者拆得太粗
- **多于 5 步**：功能可能太大了，拆成两个 PR

### Step 3 — 实现最小版本（walking skeleton）

```text
现在只做第 1 步：实现骨架，保证能跑通主流程。
- 不加错误处理
- 不加 loading / empty / error 三态
- 数据可以是硬编码
```

先让骨架跑通，再往里填肉。
反过来的顺序（先写一堆 state，最后拼主流程）最容易出"组件各自能跑、串起来不行"的故障。

### Step 4 — 补边界

```text
现在补以下边界：
- loading 态
- 错误态（网络失败、权限拒绝、超时）
- 空态
- 横竖屏 / iPad 布局
```

逐个补，别一口气让它补全。

### Step 5 — 优化

能跑、边界全了之后再谈性能和代码质量。顺序不能倒。

## 流程 C：重构

### Step 1 — 给两个方案

```text
现状：<贴代码或描述>
目标：<要重构成什么样>

不要动代码。先给两个方案：
- 方案 A：保守改动（只动必要部分）
- 方案 B：彻底重构（重新设计）

每个方案说明：
- 改动范围（文件数、行数）
- 风险点
- 回滚难度
```

### Step 2 — 选保守方案

**我的默认选择是 A**。理由：

- B 方案的"好处"在 review 时很性感，在出故障时很难追
- 保守方案可以分多轮叠加，每轮都能 ship
- 大重构一旦中途发现方向错，沉没成本高

除非 A 方案明显会留下更大的技术债，否则默认 A。

### Step 3 — 分轮执行，每轮独立可 ship

每一轮改动满足：

- diff < 200 行（超过就拆）
- 测试全过
- 代码 ship 出去不会引入新问题（哪怕下一轮不做了也是一个可用状态）

### Step 4 — 每轮验证

```bash
# 测试 + 性能基准（如果是性能重构）
xcodebuild test -scheme MyApp
# 如果有 snapshot test
xcodebuild test -scheme MyApp -only-testing:MyAppTests/SnapshotTests
```

---

## 我的原则

- **一轮只做一件事**：不在修 bug 的轮次里顺手加功能
- **控制改动范围**：diff 能小绝不大
- **必须验证**：跑不起来就是没做完
- **每轮总结**：把这一轮学到的写进 `CLAUDE.md` 或 memory
- **不信任"应该"**：只信任"刚才真跑过"

---

## 三个反模式

### 反模式 1：一口气跑完

```text
❌ 分析完 + 方案 + 改代码 + 跑测试 + 总结 都在一轮
```

中途出问题无法分段回滚，且对话越长 Claude 越容易丢前面的约束。
**拆成 4-5 轮，每轮有明确边界**。

### 反模式 2：不验证就进下一步

```text
❌ 写完不跑，直接进入下一个任务
```

bug 积累到最后，定位成本指数级上升。
**每步都必须跑起来才进下一步**。

### 反模式 3：让 AI 决定要不要做某件事

```text
❌ "你觉得要不要加个缓存？"
```

Claude 会根据"是否合理"来决定，但**是否该加取决于你的业务约束**（数据新鲜度、一致性要求等），
Claude 不知道这些。**决策由你出，Claude 只执行**。

---

## 一句话总结

**用 AI 执行工程流程，而不是用 AI 替代工程思考。**

流程的每一步都是你在做决策，Claude 在加速执行 —— 反过来就失控。
