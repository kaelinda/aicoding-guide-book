# 11-Claude Code 完整项目实战

> 用 Claude Code 把一个 iOS 小工具从零做到能跑，完整走一遍流程。

---

## 这篇讲什么

前 10 篇是分开讲技巧（上下文 / 记忆 / Skills / 多 Agent …），
这篇把它们串起来：**真做一个小 app，看流程怎么跑，不是看 prompt 怎么写**。

示例项目：**极简番茄钟**（SwiftUI）。
选它是因为：

- 功能边界清晰（计时器 + 两三个状态 + 一点持久化）
- 能覆盖"状态 / 生命周期 / 持久化 / UI"四类 iOS 典型问题
- 一个下午能跑起来，不会陷入细节

---

## Step 1：先定义任务，再开会话

新开 Claude Code 会话前，先把目标写清楚贴进去：

```text
目标：做一个 SwiftUI 番茄钟 app。

必须功能：
- 25 分钟倒计时，开始 / 暂停 / 重置
- 结束时给一次本地通知
- 当天完成次数记录（重启 app 不丢）

不做：
- 多主题、统计图表、iCloud 同步、订阅
- 支持 iOS 17 及以上即可，不兼容老系统
```

为什么一上来就写这个：**没写下来的边界 = Claude 会默认帮你加**。
番茄钟很容易被加上"长休息 / 短休息 / 自定义时长 / 专注分类"等一堆没要的功能。

---

## Step 2：让 Claude 先看项目，不要动手

```text
先不要改代码：
1. 列出可行的项目结构（目录 / 文件 / 主要类型）
2. 指出最容易踩坑的地方（状态 / 生命周期 / 后台计时）
3. 最后给出 3-5 步的拆分方案
```

Claude 大致会回：

- `TimerEngine`（纯逻辑，可测）+ `TimerView`（UI）分层
- 风险点：app 进后台时 `Timer` 会停，要用 `Date` 存起点
- 拆分：①核心引擎 → ②UI 绑定 → ③通知 → ④持久化

这一步的产出要你**读一遍、圈出不同意的**。比如 Claude 给了 MVVM 但你想用 `@Observable`，
就这时就说："用 `@Observable`（iOS 17+），不要 `ObservableObject`"。

不要等它写完再返工。

---

## Step 3：拆任务，逐步批准

让 Claude 把 Step 2 的拆分方案落成可执行清单：

```text
把刚才的 4 步拆成具体任务，每步给：
- 产出文件
- 验收标准（怎么算这步做完）
- 预估改动行数
```

得到类似：

| 步骤 | 产出 | 验收 |
|------|------|------|
| ① 引擎 | `TimerEngine.swift` | `TimerEngineTests` 5 个用例全过 |
| ② UI | `TimerView.swift` | Preview 里能 Start/Pause/Reset |
| ③ 通知 | `NotificationService.swift` + Info.plist 权限 | 模拟器收到一条本地通知 |
| ④ 持久 | `StatsStore.swift` | 杀进程再打开，今日计数还在 |

**关键动作**：到这里先停下。
手动把这张表复制到 `CLAUDE.md` 或会话顶部置顶，作为后面每一步的对照基准。

---

## Step 4：一步一步实现，每步都验证

每一步都是独立的一轮对话：

```text
现在只做第 ① 步：TimerEngine。
- 不要写 UI
- 不要写持久化
- 写完附带 XCTest 用例
```

写完立刻：

```bash
xcodebuild test -scheme Pomodoro -destination 'platform=iOS Simulator,name=iPhone 15'
```

**测试没过别进下一步**。这是整个流程里最容易偷懒的地方，
但也是"完整项目实战"和"拼出一堆跑不起来的代码"的分水岭。

第 ③ 步的通知权限会卡一下 —— Info.plist 要加 `NSUserNotificationsUsageDescription`。
Claude 如果只写了 `requestAuthorization` 不改 Info.plist，在模拟器会静默失败。

---

## Step 5：整体跑一遍，补边界

四步做完先整体跑，然后针对 iOS 特有的边界问题单独 review：

```text
现在 review 整体代码，重点检查：
- app 进后台 10 分钟再回前台，计时是否准
- 横竖屏切换时状态是否丢
- 点通知进来是否会重复计数
- 重复点 Start 按钮会不会起两个 Timer
```

这一步基本会揪出 2-3 个问题。
番茄钟这种"看起来简单"的 app，前台 `Timer` + 后台时间差的处理是经典坑。

---

## Step 6：总结沉淀，进下一个项目

收尾两件事：

1. **更新 `CLAUDE.md`**：把这次踩的坑记下来（`Timer` 用 `Date` 锚点、通知权限要 plist）
2. **记一条 memory**：`/memory add` 把"番茄钟后台计时的正确做法"沉淀下来，下次做计时类功能直接调

如果你有 Skill 习惯，可以把 **"iOS SwiftUI 新项目脚手架"** 做成一个 Skill（参见 `05-ClaudeCode常用Skills.md`）。
下次开新项目直接 `/skill ios-swiftui-new`，省掉 Step 1-2 的重复对话。

---

## 这个流程里最容易偷懒的 3 个地方

1. **跳过 Step 1 直接说"做个番茄钟"**：出来的东西 80% 能用，剩下 20% 不是你要的那 20%
2. **跳过 Step 3 的任务表**：没有对照基准，中途发现跑偏也不知道从哪一步偏的
3. **Step 4 每一步不验证**：等全部写完再跑，失败时定位困难、回滚成本高

---

## 什么项目适合这个流程

✅ 适合：

- 有明确边界的新 app / 新模块（1-3 天的量）
- 熟悉的领域，但你想试试 AI 能不能提速
- 练习沉淀 `CLAUDE.md` 和 memory 的项目

❌ 不适合：

- 探索性的"我也不知道要做成什么样"的项目 —— 先用 `/brainstorm` 或白板想清楚再开工
- 改动散在十几个模块的大重构 —— 走大师篇的多 Agent 流程
- 一次性脚本（写完就删）—— Step 1-2 不划算，直接写

---

## 一句话总结

**"完整项目实战"不是"让 AI 写完一个 app"，是"让你用 AI 把一个工程流程跑稳"。**

写代码的部分 Claude 比你快，但定义边界、拆任务、每步验证、沉淀经验 ——
这几件事你不主导，就没有"完整"，只有一堆散落的代码片段。
