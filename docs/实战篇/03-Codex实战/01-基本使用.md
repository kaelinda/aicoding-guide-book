## 启动交互模式

```bash
cd ~/Projects/my-app
codex
```

```text
╭─────────────────────────────────────╮
│ Codex CLI                            │
│ Model: gpt-5-codex                   │
│ Workspace: ~/Projects/my-app         │
╰─────────────────────────────────────╯

What would you like to build?
```

> 📌 模型名 / 命令名以 Codex 当前版本为准，下面示例是常见默认。

## 三个核心斜杠命令

### `/help`

```text
/help
```

列出所有斜杠命令。**忘了就敲它**，不要去翻文档。

### `/model`

切换模型：

```text
/model gpt-5-codex       # Codex 专用（默认，编码优化）
/model gpt-5.4           # 最强推理
/model gpt-5.3-codex     # 速度 / 质量平衡
```

**经验**：

- 写普通代码 / Refactor：`gpt-5-codex`
- 架构决策 / 难 bug 排查：临时切到 `gpt-5.4`
- 想更快 / 省 token：`gpt-5.3-codex`

### `/reasoning`

控制推理深度（会影响耗时和质量）：

```text
/reasoning high     # 深度推理，复杂任务
/reasoning medium   # 平衡模式
/reasoning low      # 快速响应
```

> ⚠️ high 模式单次请求可能 30s+ 才返回，且 token 消耗大。**不要默认开 high**，遇到复杂问题临时切一下用完再切回来。

## 基础实战案例

### 案例 1：创建一个 Swift 文件

```text
你：创建一个网络请求管理器，支持：
    - GET/POST 请求
    - 错误处理（自定义 NetworkError）
    - Token 管理（从 Keychain 取）
    使用 async/await，iOS 15+

Codex：
[分析需求] [创建 NetworkManager.swift]

已创建：Sources/NetworkManager.swift

包含方法：
- get<T: Decodable>(url:) async throws -> T
- post<T: Decodable>(url:, body:) async throws -> T
- setToken(_:) / clearToken()
- private func handleError(_:) -> NetworkError

是否需要添加单元测试？ [Y/n]
```

**review 要点**：

- 它是不是真从 Keychain 读 token，还是硬写了个 UserDefaults
- `NetworkError` 是不是**新建**的，会不会和项目已有类型冲突
- 响应 4xx 时抛异常还是返回值，和项目约定是否一致

### 案例 2：代码审查

```text
你：审查 UserViewModel.swift 的代码质量

Codex：
[读取文件] [分析代码结构]

代码审查报告：

✅ 优点：
- 使用 async/await 现代语法
- 错误处理完善
- 遵循 MVVM 架构

⚠️ 建议改进：
1. 第 34 行：考虑用 @MainActor 标记
2. 第 67 行：HTTP 请求前该先判 Reachability
3. 第 89 行：魔法数字 300 抽成常量

❌ 问题：
- 第 45 行：潜在的内存泄漏
  原因：Task { } 里未使用 [weak self]
  修复：添加 [weak self] 捕获列表

是否自动修复？ [Y/n]
```

**强烈建议**：这种一键 "Y" 的诱惑要抵抗。把 review 和修复**拆成两轮**：

```text
# 第二轮
针对上一轮的"第 45 行内存泄漏"，给最小 patch，只改这一处。
```

### 案例 3：重构代码

```text
你：把 UserService 重构成 Repository 模式。
约束：
- 不动外部调用点（UserService 对外 API 保持不变）
- 分两步：先抽 protocol，再换实现
- 每步之间让我 review 再继续

Codex：
[分析现有代码] [规划重构步骤]

重构计划：
Phase 1（本轮）：
  1. 创建 UserRepository 协议
  2. 把 UserService 的数据访问抽到 protocol

Phase 2（下一轮）：
  3. 实现 RemoteUserRepository（HTTP）
  4. 实现 LocalUserRepository（Keychain/FileSystem）
  5. UserService 改为持有 Repository 对象

现在执行 Phase 1...
[改动 UserService.swift + 新增 UserRepository.swift]

✅ Phase 1 完成。建议你先跑一次测试再进 Phase 2。
```

**关键手势**：**在 prompt 里明确分阶段 + 让我 review**。Codex 如果没约束，会一把梭改 10 个文件，出错了回滚都难。

## 交互模式里的几个实用技巧

- **多行输入**：直接粘贴带换行的 prompt，Codex 会正确识别
- **引用文件**：用 `@filename.swift` 显式把文件拉进上下文
- **中途打断**：`Esc` 或 `Ctrl+C` 停止当前生成
- **清空上下文**：`/clear` 开新对话（别把一个 session 拖太长）
- **看用量**：`/cost` 查当前 session 估算消耗
