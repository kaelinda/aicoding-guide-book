## 1. `AGENTS.md` — Codex 的项目规则

在项目根目录建 `AGENTS.md`，Codex 启动时会自动读取。作用类似于 Claude Code 的 `CLAUDE.md` 或 Cursor 的 `.cursorrules`。

### 最小模板

```markdown
# <项目名> AGENTS 配置

## 技术栈
- Swift 5.9+
- SwiftUI（@Observable / iOS 17+）
- Swift Testing（不是 XCTest）
- 网络层：URLSession + async/await，不用 Combine

## 目录
- Presentation/Views/     — SwiftUI View
- Presentation/ViewModels/ — @Observable store
- Domain/                 — 领域模型 + UseCase
- Data/                   — Repository / DataSource

## 编码规范
- 4 空格缩进；camelCase 函数，PascalCase 类型
- Public API 必有 /// 文档注释
- 禁用强制解包 `!`（IBOutlet 除外）

## 禁止事项
- 不要在 View 里发网络请求
- 不要新建 Combine Publisher
- 不要把 token 存 UserDefaults
- 涉及 Keychain / StoreKit / Privacy Manifest 的改动必须让我 review
```

Codex 会把这份文件当作"项目级系统提示"，每次会话都生效。

### 经验

- **保持 50-80 行**：太长塞爆上下文
- **禁令 + 正面指引 1:2**：别只说"不许"
- **入 git，团队共享**

> 📌 `AGENTS.md` 的命名和加载规则以 Codex [官方文档](https://developers.openai.com/codex) 为准，不同版本可能略有差异。

## 2. 子代理（Subagents）

Codex 支持并行派发多个子任务。

### 基本用法

```bash
# 在交互模式里
你：并行重构 User 模块和 Product 模块。
    - 先各自做到"只抽 protocol"这步
    - 不要动外部调用点
    - 做完后我再决定合并

Codex：
启动 2 个 subagent...
[Agent 1: 重构 User 模块]
[Agent 2: 重构 Product 模块]

Agent 1 完成 ✅（改动 User/UserService.swift + 新增 UserRepository.swift）
Agent 2 完成 ✅（改动 Product/ProductService.swift + 新增 ProductRepository.swift）

合并结果：无冲突
```

### 适合的场景

- **互不相关的任务并行**（重构 User 模块 + Product 模块）
- **并行扫描 / 审查**（同时让 3 个 subagent 看 3 个目录）
- **独立的验证任务**（跑测试 + 跑 lint + 生成报告）

### 不适合的场景

- **有依赖的任务**（"先重构再测试"必须顺序，并行没意义）
- **需要共享上下文的决策**（架构讨论别拆）
- **简单任务**（起 subagent 的开销 > 省下的时间）

和 Claude Code 的 Task/subagent 机制原理相通，详见 [进阶篇/08-01-Task 工具与 Subagent](../../进阶篇/08-ClaudeCode新功能/01-Task工具与Subagent.md)。

## 3. 网络搜索

Codex 自带网络搜索工具，可以查最新的 API、框架变更。

```text
你：搜索 SwiftUI 在 iOS 17-18 之间的动画 API 变化并给出示例。
不要依赖你的训练数据，走网络搜索。

Codex：
[调用网络搜索]
[汇总官方文档 + WWDC session]

SwiftUI 动画 API 变化要点：
1. `.animation(_, value:)` —— iOS 15+（带 value 参数，推荐）
2. `withAnimation { } ` —— 编程式触发
3. `.transition(...)` + `.id(...)` —— 视图切换动画
4. iOS 18 新增的 `@Animatable` 宏和 `Animation.spring(response:...)` 参数

示例代码：
[生成可运行的 SwiftUI 片段]

来源：
- developer.apple.com/documentation/swiftui/animation
- WWDC 2024 "What's new in SwiftUI"
```

### 什么时候该强制网络搜索

- 涉及 **2024-2026 年新 API**（训练数据可能过时）
- 涉及**第三方库版本迁移**（如 Kingfisher 7 → 8）
- 不确定 AI 是不是在幻觉 API

### 使用建议

- **明确说"走网络搜索、不要依赖训练数据"**
- **要求溯源**："给官方文档链接 / WWDC session 号"
- **交叉验证**：搜完让它**用 Context7 MCP 再查一次**（见 [进阶篇/02-MCP 协议 § iOS 清单](../../进阶篇/02-MCP协议/README.md#-ios-开发者的-mcp-实用清单)）

## 4. CI / CD 集成

### 非交互模式

```bash
# 一次性任务，不进交互模式
codex --non-interactive \
      --task "审查 PR 的代码质量" \
      --output review.md
```

### GitHub Actions 示例

```yaml
# .github/workflows/codex-review.yml
name: AI Code Review

on: [pull_request]

jobs:
  review:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Setup Node
        uses: actions/setup-node@v4
        with:
          node-version: 20

      - name: Install Codex
        run: npm install -g @openai/codex

      - name: Codex Review
        env:
          OPENAI_API_KEY: ${{ secrets.OPENAI_API_KEY }}
        run: |
          codex --non-interactive \
                --task "审查本 PR 的 diff，重点看 SwiftUI 状态管理和并发安全" \
                --output review.md

      - name: Post Review to PR
        uses: actions/github-script@v7
        with:
          script: |
            const fs = require('fs');
            const review = fs.readFileSync('review.md', 'utf8');
            github.rest.issues.createComment({
              issue_number: context.issue.number,
              owner: context.repo.owner,
              repo: context.repo.repo,
              body: review
            });
```

### 使用建议

- **Secrets 用 `OPENAI_API_KEY`** 而不是 ChatGPT 登录态（CI 里没有浏览器）
- **任务要具体**（别写 "审查代码"，要写"审查 diff、重点看 X / Y"）
- **设 token 预算**：CI 上跑容易失控，Codex 非交互模式通常可带 `--max-tokens` 限额
- **结果 post 成 PR comment**，而不是挡 merge —— 给人看，不是给 bot 决定

> ⚠️ CI 自动点评 **不应**直接挡 merge，更不该自动改代码。AI review 会漏掉真问题、也会误报。**人类仍然是最后把关人**。
