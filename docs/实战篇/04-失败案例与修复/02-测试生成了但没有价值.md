# 02-测试生成了但没有价值

> 有测试文件，不代表真的有测试价值

## 🎯 这个问题为什么常见

很多人在让 AI 生成测试后，会很容易获得一种安心感：

- 生成了很多测试函数
- 文件名也很规范
- 甚至测试都能通过

于是就会觉得：

> “测试已经补齐了。”

但真实情况常常不是这样。

AI 很容易生成的是：**形式上完整、实际上覆盖不足** 的测试。

## 🤔 什么叫“没有价值的测试”

下面这些都属于常见的低价值测试：

- 只测最理想流程
- 只测很浅的 getter / setter
- 断言过于宽松
- 没覆盖边界情况
- 没覆盖真正容易出错的地方
- 测试看起来很多，但没有保护关键逻辑

换句话说：

> 测试数量很多，不等于它们真的能帮你挡住 bug。

## 🐞 一个典型场景

假设你有一个 `TodoListViewModel`，支持：

- 添加事项
- 删除事项
- 切换完成状态
- 搜索过滤

你让 AI 帮你生成测试，它可能给你：

- `testAddTodo()`
- `testDeleteTodo()`
- `testToggleCompletion()`

看起来没什么问题。

但真正的问题是：

- 它只测了最简单的数据
- 没测搜索后删除
- 没测空输入
- 没测重复操作
- 没测状态同步

这就是典型的“测试文件有了，但价值不够”。

## ❌ 错误 Prompt 示例

```text
请帮我为这个 ViewModel 生成完整测试。
```

这个问题太宽了，AI 很容易默认：

- 先把最表面的 happy path 测一遍
- 只要测试能跑通就算完成

但这不等于测试真正覆盖了业务风险。

## 🔍 你应该怎么判断测试有没有价值

建议重点看下面 4 个问题。

### 1. 它测的是“功能风险”，还是“表面存在”

例如：

- 测添加事项数量变化，这算基础
- 但有没有测“搜索状态下删除是否正确”
- 有没有测“边界值输入是否安全”

真正有价值的测试，应该更靠近风险点。

### 2. 它有没有覆盖边界情况

至少要问：

- 空输入怎么办
- 空列表怎么办
- 重复点击怎么办
- 只有一条数据时怎么办
- 搜索后操作怎么办

如果这些都没测，测试保护力就会比较弱。

### 3. 断言是不是太松

例如：

- 只断言“不为空”
- 只断言“数量大于 0”
- 只断言“方法执行了”

这类断言往往很难真正发现逻辑错误。

### 4. 测试名和测试行为是否一致

有时候测试名看着很专业，但实际断言根本没覆盖标题里说的场景。

所以不能只看函数名，要看里面到底测了什么。

## ✅ 更好的 Prompt 写法

```text
请帮我为这个模块补测试，但不要只生成表面测试。

当前模块的核心功能：
- 添加事项
- 删除事项
- 切换完成状态
- 搜索过滤

请先做这 4 件事：
1. 列出最值得优先测试的风险点
2. 区分 happy path、边界情况、错误情况
3. 给出推荐测试清单
4. 再生成测试代码

要求：
- 优先覆盖关键业务逻辑
- 不要只测简单 getter/setter
- 请说明每个测试在保护什么风险
```

这类问法会比“帮我生成完整测试”稳定得多。

## 🛠 修正思路

如果你已经拿到一份“低价值测试”，不要急着全部删掉。

可以按这个顺序修：

### 第一步：标出已有测试覆盖了什么

让 AI 先解释：

```text
请分析当前这些测试分别覆盖了什么，没有覆盖什么。
```

### 第二步：补风险点，而不是补数量

```text
请不要新增很多测试数量。
请优先补目前最缺失的高风险场景。
```

### 第三步：加强断言

```text
请检查这些测试的断言是否过于宽松，并给出更严格的断言建议。
```

## ✅ 一份更靠谱的验收清单

当你评估一组测试是否“有价值”时，可以至少看这些：

- [ ] 是否覆盖核心业务逻辑
- [ ] 是否覆盖边界情况
- [ ] 是否覆盖错误情况
- [ ] 断言是否足够具体
- [ ] 测试名是否和测试内容一致
- [ ] 是否能真正阻止关键回归

## 💡 这个案例真正要学会什么

### 1. 测试的存在不等于测试的价值
真正重要的是它挡不挡得住问题。

### 2. AI 默认更容易先生成 happy path
你必须主动引导它补边界和风险。

### 3. 测试生成前先定测试范围，通常会更稳
这是比“直接写测试代码”更重要的一步。

## ✅ 小结

低价值测试的问题，不在于“写错了”，而在于：

- 看起来很多
- 实际保护很弱
- 容易让人误以为已经安全

记住这句话：

> **有测试，不代表有保护；只有覆盖风险的测试，才真正有价值。**
