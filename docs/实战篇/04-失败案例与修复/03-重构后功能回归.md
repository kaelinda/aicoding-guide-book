# 03-重构后功能回归

> 代码结构变漂亮了，不代表功能一定还正常

## 🎯 为什么这个问题很常见

AI 很擅长提出更“整洁”的结构方案。

例如：

- 把一个大 View 拆成多个子视图
- 把业务逻辑提到 ViewModel
- 把数据层抽成 Service / Repository
- 统一命名和目录结构

这些方向本身没有问题。

问题在于：

> **重构的目标是“保持行为一致，改善结构”，不是“为了结构更漂亮，顺便改掉原来的行为”。**

而 AI 在做重构时，很容易出现“结构变好了，但行为悄悄变了”的情况。

## 🐞 一个典型场景

假设你有一个 Todo 列表页面，原本支持：

- 新增事项
- 删除事项
- 搜索过滤
- 切换完成状态

你让 AI 帮你做重构，目标是：

- 拆出子视图
- 整理状态管理
- 提高可读性

结果它做完后：

- 页面看起来更清爽了
- 文件结构也更整齐了
- 但是搜索结果更新变慢
- 删除后某些状态不同步
- 某个按钮点击行为和之前不一样了

这就是典型的“功能回归”。

## 🤔 为什么重构后最容易回归

通常有这几个原因：

### 1. AI 把“结构优化”顺手做成了“逻辑调整”
很多时候你只想整理代码，但 AI 会在重构过程中一起：

- 改状态流
- 改数据来源
- 改默认值
- 改交互顺序

结果就是：表面上是重构，实际上已经触碰了行为层。

### 2. 重构范围太大
如果一次让 AI：

- 拆文件
- 改命名
- 改状态结构
- 改业务逻辑
- 再顺手做优化

那你几乎很难知道回归到底是哪一步造成的。

### 3. 没有“重构前后的行为基线”
如果你在重构前没有明确列出：

- 哪些功能必须保持一致
- 哪些交互不能变
- 哪些状态必须保持原逻辑

那么重构后很容易陷入“看起来差不多就算一样”的误判。

## ❌ 错误 Prompt 示例

```text
帮我把这段代码重构得更优雅一点。
```

这个问题太模糊了，AI 很可能默认：

- 怎么看着更专业就怎么改
- 怎么看着更整洁就怎么拆

但“更优雅”并不是一个可验证的目标。

## ✅ 更好的 Prompt 写法

```text
我想重构这段代码，但请严格遵守下面的约束：
- 当前功能行为必须保持一致
- 这次只做结构整理，不新增功能，不改变交互
- 请优先做最小可验证重构
- 如果某一步会影响行为，请先明确指出风险

请先做这 4 件事：
1. 列出当前代码的主要结构问题
2. 给出最小可行重构方案
3. 标明哪些行为必须保持完全一致
4. 给出重构后的回归检查清单
```

这种问法的关键点是：

- 先定义边界
- 明确“不许顺手改行为”
- 要求它给出回归检查清单

## 🔍 你应该如何发现“功能回归”

建议重点看下面 4 类变化：

### 1. 交互行为有没有变化
例如：
- 原来点一下就更新，现在要点两下
- 原来删除后立即刷新，现在延迟刷新
- 原来搜索即时生效，现在需要额外操作

### 2. 状态同步有没有变化
例如：
- 列表更新了，但计数没更新
- UI 更新了，但数据源没同步
- 某个子视图状态和父视图状态脱节

### 3. 边界情况有没有变弱
例如：
- 原来空输入有保护，重构后没了
- 原来错误提示完整，重构后被漏掉

### 4. 可测试行为有没有变得更难验证
如果重构后看起来更抽象了，但反而更难判断行为是否正确，这也是一个危险信号。

## 🛠 修正思路

如果你已经发生了“重构后回归”，建议按这个顺序处理：

### 第一步：先列出回归现象

```text
请先不要继续重构。
先帮我列出当前重构后与原行为不一致的地方。
```

### 第二步：找出最可能的改动点

```text
请分析这次重构中，最可能导致行为变化的是哪几处改动。
```

### 第三步：做最小回退或最小修复

```text
请优先用最小修改恢复原行为，不要再继续顺手优化。
```

### 第四步：补回归检查清单

```text
请为这次重构补一份回归检查清单，确保后续再改时不会重复踩坑。
```

## ✅ 一份更靠谱的回归检查清单

当你做完一次重构后，至少检查这些：

- [ ] 原有核心功能是否都还能正常工作
- [ ] 原有交互顺序是否保持一致
- [ ] 原有边界处理是否保持一致
- [ ] UI 状态与数据状态是否仍然同步
- [ ] 重构后是否更容易继续修改，而不是更难理解
- [ ] diff 是否控制在合理范围

## 💡 这个案例真正要学会什么

### 1. 重构的成功标准不是“更漂亮”，而是“更稳”
如果结构更优雅，但行为变了，那这次重构就是有问题的。

### 2. 重构前要先定义“哪些行为不能变”
这是最容易被忽略、但最重要的一步。

### 3. 小步重构永远比一步到位更安全
一旦改动太大，回归问题就很难定位。

## ✅ 小结

“重构后功能回归”是 AI 协作里很典型的一类问题，因为 AI 特别容易把“整理代码”扩展成“改动行为”。

记住这句话：

> **重构的价值，不在于代码更像教材，而在于功能不变的前提下更容易维护。**
