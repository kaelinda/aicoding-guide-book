# 04-失败案例与修复

> AI 会生成很多“看起来对、实际上不对”的内容，本章教你怎么纠偏

## 🎯 为什么这一章很重要

很多 AI 编程教程只展示成功案例，但真实开发里，真正拉开差距的往往不是“第一次就写对”，而是：

- 你能不能快速发现问题
- 你能不能判断问题出在哪里
- 你能不能让 AI 在原有结果上做最小修复
- 你能不能从一次失败里总结出更好的协作方式

这就是这一章存在的意义。

## 🤔 AI 编程里常见的失败，不是只有报错

很多初学者会把“失败”理解成编译报错。

但实际上，AI 更常见的问题包括：

- **代码能编译，但逻辑不对**
- **页面能显示，但状态管理混乱**
- **结果看起来像对的，其实边界情况没处理**
- **重构后结构更漂亮了，但功能回归了**
- **测试生成出来了，但并没有真正覆盖关键场景**

所以，本章重点不是教你“修语法”，而是教你建立更稳定的问题判断力。

## 📦 本章会覆盖什么

本章会逐步补充这类高频问题：

### 1. 代码能跑，但逻辑错误
最常见，也最危险。

入口：
- [01-代码能跑但逻辑错误](./01-代码能跑但逻辑错误.md)

### 2. UI 正常，但状态流转混乱
比如：
- 页面显示了
- 按钮也能点
- 但刷新后状态不一致
- 或多个状态互相覆盖

### 3. 测试生成了，但没有实际价值
比如：
- 只测 happy path
- 没测边界条件
- 断言过于宽松

### 4. 重构后功能回归
比如：
- 代码结构更优雅了
- 但某些交互悄悄坏掉了

## 💡 处理 AI 错误时的 3 条原则

### 原则 1：不要一出错就全部重来
很多时候，最好的方式不是“重新生成一版”，而是：

- 保留当前结果
- 明确指出问题
- 要求做最小修改

这样更容易保住已经正确的部分。

### 原则 2：先描述现象，再要求修复
与其说：

```text
帮我修一下
```

不如说：

```text
当前现象是：
- 页面能显示
- 点击删除后删错了对象
- 搜索结果和原列表不同步

请先分析原因，再给我最小修改方案。
```

### 原则 3：失败案例要复盘，不只是修完就算
每次失败都值得问自己：

- 是需求没讲清楚？
- 是 AI 一次做太多？
- 是我没有及时验收？
- 还是我自己也没想清楚状态流？

只有复盘，失败才会变成经验。

## ✅ 小结

真正会用 AI 的人，不是从不出错的人，而是：

- 出错后能快速定位
- 能把问题讲清楚
- 能引导 AI 做出更稳定的修复

这就是从“会用工具”走向“会协作开发”的关键一步。
