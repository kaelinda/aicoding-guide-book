# 02-ViewModel 拆分

> 当功能越来越多时，如何让 AI 帮你整理状态和职责

## 🎯 这个场景解决什么问题

很多初级 iOS 开发者在功能变多以后，最常见的问题是：

- ViewModel 越来越长
- 状态越来越多
- 搜索、排序、删除等逻辑混在一起
- 自己越来越不敢改

这时候，AI 最适合帮你的，是做一次更克制的职责整理。

## 🤔 什么时候说明你该拆 ViewModel 了

常见信号包括：

- 一个 ViewModel 同时负责 UI 状态、业务逻辑、网络请求
- 一个文件越来越长
- 新增一个功能就要改很多地方
- 自己很难说清楚“谁负责什么”

## ✅ 推荐工作流

### 第一步：先识别问题
先让 AI 帮你总结：

- 当前 ViewModel 有哪些职责
- 哪些职责耦合在一起
- 哪些问题最值得优先处理

### 第二步：只做最小拆分
更稳的做法通常是：

- 提取搜索逻辑
- 提取过滤 / 排序逻辑
- 提取网络请求服务
- 提取格式化逻辑

不要一口气把简单项目拆成很复杂的架构。

### 第三步：保持行为不变
拆分时最重要的约束是：

- 不改变现有交互行为
- 不顺手大改状态流
- 不一次改太多文件

### 第四步：补回归检查
拆完后必须检查：

- 原有功能是否正常
- 状态切换是否仍然一致
- 搜索 / 删除 / 更新有没有回归

## 💬 推荐 Prompt

```text
请帮我分析当前 ViewModel 是否需要拆分。

项目背景：
- 技术栈：SwiftUI
- 当前模块：{ViewModel 名称}
- 当前主要问题：{例如 文件太长 / 逻辑混在一起 / 不好维护}

请按下面顺序协助我：
1. 总结当前职责
2. 指出最值得优先拆开的部分
3. 给出最小可行拆分方案
4. 说明重构后应该如何验证行为没有变化

要求：
- 优先小步拆分
- 不要过度设计
- 保持现有功能行为一致
```

## ⚠️ 最容易踩的坑

### 坑 1：把“拆 ViewModel”理解成“必须上复杂架构”
很多时候，最好的拆分只是：

- 职责更清楚一点
- 文件更短一点
- 状态更容易理解一点

### 坑 2：拆结构时顺手改逻辑
这是最容易引发功能回归的地方。

### 坑 3：为了“看起来专业”而拆过头
如果拆完以后理解成本更高，那这次拆分未必有价值。

## ✅ 验收清单

- [ ] ViewModel 职责比之前更清晰
- [ ] 原有功能行为保持一致
- [ ] 主要状态流还能解释清楚
- [ ] 文件组织更利于继续维护
- [ ] 没有明显过度设计

## ✅ 小结

ViewModel 拆分的目标不是层次更多，而是职责更清楚。
