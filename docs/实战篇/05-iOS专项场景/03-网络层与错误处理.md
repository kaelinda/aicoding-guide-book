# 03-网络层与错误处理

> 从静态页面走向真实数据交互时，AI 最容易帮上忙，也最容易埋坑

## 🎯 这个场景解决什么问题

当你开始接真实数据时，复杂度会明显上升。

常见问题包括：

- 请求逻辑写在哪里
- 错误状态怎么展示
- loading 怎么管理
- mock 数据和真实接口怎么切换
- 页面、ViewModel、Service 的职责如何划分

这是 iOS 开发里非常典型的一步，也是 AI 很适合参与的一步。

## ✅ 推荐工作流

### 第一步：先用 mock 数据跑通
先不要急着接真实接口。

先让 AI 帮你完成：

- 页面结构
- ViewModel 状态
- loading / error / data 三态
- 模拟请求流程

### 第二步：再接真实接口
当 mock 版本稳定后，再继续：

- 定义 Service
- 定义数据模型
- 解析返回数据
- 处理错误

### 第三步：最后再整理结构
例如：

- 提取 NetworkService
- 统一错误模型
- 整理请求入口

## 💬 推荐 Prompt

```text
请帮我设计一个适合初学者理解的 iOS 网络请求结构。

项目背景：
- 技术栈：SwiftUI
- 当前页面：{例如 天气查询页 / 用户列表页}
- 目标：支持 loading、成功、失败 三种状态

请按下面顺序协助我：
1. 先给出最小可行结构
2. 先用 mock 数据跑通
3. 再说明如何替换成真实接口
4. 最后说明错误处理和验证方式

要求：
- 保持结构清晰
- 不要过度设计
- 适合初学者理解
```

## ⚠️ 最容易踩的坑

### 坑 1：还没理清状态就接真实接口
这样最容易把 UI、状态和请求混在一起。

### 坑 2：没有明确 loading / error / data 三态
如果这三态不清楚，页面逻辑通常会越来越乱。

### 坑 3：错误处理只做打印，不做用户可见反馈
这是很常见的“看起来能跑，体验很差”的问题。

## ✅ 验收清单

- [ ] 页面能显示 loading 状态
- [ ] 请求成功时结果展示正确
- [ ] 请求失败时有明确错误提示
- [ ] mock 和真实接口切换路径清晰
- [ ] 页面、ViewModel、Service 职责基本明确

## ✅ 小结

网络层与错误处理的关键，不是写出最复杂的封装，而是先把数据流和状态流做清楚。
