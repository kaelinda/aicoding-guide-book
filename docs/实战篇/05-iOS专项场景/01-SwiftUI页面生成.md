# 01-SwiftUI 页面生成

> 用 AI 快速搭页面，但不要把“能显示”误当成“已经做好”

## 🎯 这个场景解决什么问题

对于 iOS 初学者来说，最容易从 AI 中立刻获得帮助的，就是页面生成。

例如：

- 列表页
- 设置页
- 表单页
- 空状态页
- 详情页

这些页面往往有比较明确的 UI 结构，非常适合先让 AI 帮你搭出第一版骨架。

## 🤔 为什么这个场景特别适合 AI

因为它通常具备这些特点：

- 页面结构相对直观
- 组件层次容易描述
- 可视化结果容易验证
- 小步迭代很自然

也就是说，它非常适合做“先生成骨架 → 再逐步细化”的协作方式。

## ✅ 推荐工作流

### 第一步：先描述页面目标
不要一上来就要求完整实现所有交互。

先说明：

- 页面用途
- 页面包含哪些区域
- 主要交互是什么
- 暂时不做什么

例如：

```text
请帮我创建一个 SwiftUI 设置页，包含：
- 用户头像区域
- 两个开关：深色模式、通知提醒
- 一个“关于我们”入口
- 风格尽量简洁
- 先只做页面骨架，不做真实数据持久化
```

### 第二步：先做静态页面
静态版本有两个好处：

- 你更容易确认布局是否符合预期
- 不会太早把状态和逻辑混进去

### 第三步：再逐步加交互
例如再补：

- 开关状态
- 导航跳转
- 表单校验
- 加载状态

### 第四步：最后再整理结构
当页面初步成形后，再让 AI 帮你：

- 提取子组件
- 调整命名
- 整理样式

## 💬 推荐 Prompt

```text
请帮我创建一个 SwiftUI 页面，要求：
- 页面类型：{例如 设置页 / 详情页 / 表单页}
- 包含区域：{列出主要区域}
- 主要交互：{列出 1~3 个核心交互}
- 先做静态或最小可运行版本
- 代码适合初学者理解
- 如果页面稍复杂，请先拆成实现步骤
```

## ⚠️ 最容易踩的坑

### 坑 1：一上来就要求“完整页面 + 完整逻辑”
这样最容易得到一个很大、很乱、自己难以理解的第一版。

### 坑 2：只看页面长得像不像
页面生成最容易让人只关注 UI，但更重要的是：

- 状态是不是清晰
- 结构是不是好维护
- 后续加逻辑会不会很乱

### 坑 3：过早组件化
组件化当然重要，但如果页面目标都没确认清楚，过早拆太多子 View 反而增加理解成本。

## ✅ 验收清单

- [ ] 页面结构符合目标
- [ ] 主要区域完整
- [ ] 交互入口清晰
- [ ] 代码没有明显过度设计
- [ ] 你能看懂页面层级
- [ ] 后续继续加逻辑不会太痛苦

## ✅ 小结

SwiftUI 页面生成是最适合用 AI 建立信心的 iOS 场景之一。

但真正高质量的用法不是“让 AI 一次把页面做完”，而是：

- 先做页面骨架
- 再逐步补交互
- 最后再整理结构

记住这句话：

> **页面生成最好的第一版，不是最完整的，而是最容易继续改的。**
