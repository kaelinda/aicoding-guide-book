# 03-Cursor 高级技巧

> `.cursorrules` / 多文件批量编辑 / 代码审查 / 文档生成。从"AI 辅助打字"升到"AI 辅助工程"。

---

## 1. `.cursorrules` — 把约束写进项目

项目根目录建 `.cursorrules` 文件，入 git 让团队共享：

```markdown
# <项目名> Cursor 规则

## 技术栈
- Swift 5.9+
- SwiftUI + @Observable（iOS 17+）
- Swift Testing（不是 XCTest）
- 第三方库：Kingfisher, swift-collections

## 架构
- MVVM，View 薄、ViewModel 持 @Observable store
- Services/ 下一律 protocol + 具体实现分离，便于测试
- Sources/Views/<Feature>/ 每个 feature 一个目录

## 编码规范
- 4 空格缩进
- 函数 camelCase，类型 PascalCase
- 每个 public 方法加 /// 文档注释
- 用 MARK: 分隔代码块

## 禁止事项
- 不要用强制解包 `!`（除 IBOutlet 和明确不可能 nil 的地方）
- 不要在 View 内直接发网络请求
- 不要新建 Combine Publisher（项目走 async/await）
- 不要在 UserDefaults 存 token / credentials（用 Keychain）
- 不要改 Services/Auth/（走单独评审）

## 输出风格
- 中文回答
- 结论先行，细节后给
- 不说"应该没问题"，结论必须附验证命令
```

Cursor 会**自动把这份规则塞进每次对话的系统提示**。相当于每次开聊都先念一遍。

### 什么该写、什么不该写

**该写**：

- 技术栈版本（最重要 —— 防止 AI 给老写法）
- 禁止事项（最省事 —— AI 会少犯错）
- 架构分层（让生成的代码放对位置）

**不该写**：

- 愿景 / 文化 / 理念（AI 用不上）
- 500 字长篇大论（塞爆上下文）
- "我希望你聪明一点"（AI 不会因此更聪明）

**经验区间**：30-80 行。超过就拆到 `docs/` 里具体文件，规则里指个路。

## 2. 多文件批量编辑（Composer — `Cmd + I`）

不再是"选中改这一段"，而是"告诉 AI 想改什么，它自己跨文件找"。

```text
你（Cmd + I）：把所有硬编码颜色（像 Color.red、UIColor(red:...)）
改成使用 DesignSystem.Colors.* 的对应条目。

DesignSystem.Colors 的定义在 Sources/DesignSystem/Colors.swift。
修改时：
- 新加的 import 只加必要的
- 不要改动其他代码
- 最后列出所有改动的文件和位置
```

### Cursor 行为

```text
[扫描所有 .swift 文件]
找到 23 处硬编码颜色：
- ContentView.swift:14, 67, 89
- ProfileView.swift:23, 45, 102, 140, 188
...

准备的 patch（已按文件分组）：
[一条条列出 diff]

Apply All / Apply One by One / Skip
```

### 关键经验

- **先让它列 plan 再 apply**：直接 Apply All 经常出意外改动
- **大型批量改动前 `git commit` 一次**：出问题能一键 revert
- **分批**：一次改 10 个文件够你 review，一次改 100 个就是盲盒
- **加明确的约束**（"只改颜色，不动其他"），否则 AI 会顺手"优化"

## 3. 代码审查模式

```text
你：审查 Sources/Features/Login/ 目录下所有文件。
维度：
- 内存泄漏（循环引用、未 [weak self]）
- 线程安全（@MainActor 漏写、Task 未 cancel）
- 状态管理（@State / @StateObject / @Observable 误用）
- 错误处理是否完整
- 测试覆盖

按 🔴 必改 / 🟡 建议 / 🟢 可忽略 分级。
不要自动修复，只出报告。
```

Cursor 产出一份 review 报告。你根据报告再决定哪些让它修、哪些手工修。

### 为什么 "不要自动修复"

AI review 自己写的代码会偏向"原谅"。**让它只挑问题不修**，保留 review / 修复的两步分离，质量更稳。

如果要修，用新的一轮对话：

```text
针对上一轮 review 的 🔴 问题 3（LoginView.swift 第 67 行内存泄漏），
给 minimal patch 修复。只改这一处，不动其他。
```

## 4. 文档注释生成

选中 Swift 函数签名，`Cmd + K` → "生成文档注释"：

```swift
/// 从服务器获取用户信息
///
/// - Parameter id: 用户唯一标识符
/// - Returns: 用户对象
/// - Throws:
///   - `NetworkError.networkUnavailable`：网络不可用
///   - `NetworkError.userNotFound`：用户不存在
///
/// ```swift
/// do {
///     let user = try await fetchUser(id: "123")
/// } catch {
///     print(error.localizedDescription)
/// }
/// ```
func fetchUser(id: String) async throws -> User {
    // ...
}
```

Xcode 会把这份注释在 Quick Help 里渲染出来。

### 踩坑

- AI 会**编造** `Throws` 的错误类型。review 时核对真实抛出的是不是这些
- 示例代码里的参数 `id: "123"` 可能不现实。看一眼实际调用链的值
