# 02-需求澄清模板

> 两种"澄清需求"的模板（新功能 / 重构）+ 不同场景下的策略选择 + 两个让 AI 帮你澄清的技巧。

---

## 模板 1：创建新功能

```markdown
## 功能描述
[一句话说清是什么]

## 用户故事
作为 [角色]，我希望 [行为]，以便 [收益]

## 技术要求
- 技术栈：[Swift / SwiftUI / SwiftData / iOS 17+]
- 架构：[MVVM / @Observable / ...]
- 数据源：[API / 本地数据库 / ...]

## UI 要求
- 布局：[描述或贴截图]
- 交互：[点击 / 滑动 / 长按 ...]
- 动画：[是否需要]

## 验收标准
- [ ] 功能 1
- [ ] 功能 2
- [ ] 测试通过

## 参考
- [类似功能的链接或截图]
- [项目里同类页面的路径]
```

### 使用建议

- **用户故事这一条经常被跳过**，但它是防止 AI 跑偏最重要的一条。写了之后 AI 会根据"收益"判断哪些是必须、哪些可以省
- **验收标准写得越具体越好**：别写"测试通过"，写"点击登录 → 调用 /auth/login → 成功跳转首页"
- 参考指路一定要具体到**项目里的文件路径**，AI 才会真去读

## 模板 2：代码重构

```markdown
## 重构目标
[为什么要重构]

## 当前代码
[贴关键文件或描述实现方式]

## 问题
- [问题 1：比如"UserViewController 1000 行，职责混乱"]
- [问题 2：比如"测试覆盖 < 20%"]

## 目标架构
[期望的架构方式，最好给一个文件结构草图]

## 约束
- 保持功能不变
- 通过现有测试
- 不破坏外部 API 兼容性

## 验收标准
- [ ] 代码结构清晰
- [ ] 测试全部通过
- [ ] 性能未降低

## 执行方式
分 N 轮，每轮独立可 ship：
- Round 1：[具体内容]
- Round 2：[具体内容]
```

### 使用建议

- **重构必须给"约束"**，特别是"保持功能不变"和"不破坏 API 兼容性"。不给 AI 会顺手改外部调用点
- **目标架构给文件结构草图**比描述更有效：
  ```
  Sources/
  ├── Presentation/
  │   ├── Views/
  │   └── ViewModels/
  ├── Domain/
  └── Data/
  ```
- **执行方式拆成多轮** —— 见 [best-practices 反面案例 2](../../best-practices.md#反面案例-2一次-prompt-要求太多功能)

## 不同场景的 Prompt 策略

Prompt 详细程度不是越高越好，要看场景。

### 场景 A：快速原型

**策略**：简化 Prompt、快速迭代。

```text
Round 1: "快速实现一个登录页原型"
Round 2: "加忘记密码入口"
Round 3: "样式优化一下，用 Material Design 风格"
```

**什么时候用**：你还不确定要什么、先看到东西再说。
**要避免**：用生产级 Prompt 做原型（浪费时间）。

### 场景 B：生产代码

**策略**：详细 Prompt、严格验收。

```text
"生产级登录页：
 1. 邮箱格式验证（RFC 5321 规范）
 2. 密码强度：≥8 位 + 大小写 + 数字
 3. 错误处理：网络异常 / 服务器错误 / 账号禁用
 4. Loading / Disabled / Error 三态
 5. 单测覆盖率 ≥ 80%（Swift Testing）
 6. 符合项目 SwiftLint 规范
 7. Public API 加文档注释"
```

**什么时候用**：真要上线的代码、团队项目。
**要避免**：快速原型阶段用这套（过度工程）。

### 场景 C：学习探索

**策略**：询问 + 验证。

```text
Round 1: "SwiftUI 中有哪些动画方式？
         请举例说明每种的适用场景"
Round 2: "哪个最适合做页面切换动画？为什么？"
Round 3: "给我一个可以直接跑的 demo"
```

**什么时候用**：学新东西 / 评估技术选型。
**要避免**：直接让 AI 写、不先理解原理。

## 让 AI 帮你澄清：两个高杠杆技巧

### 技巧 1：反向提问（让 AI 帮你想清楚要什么）

```text
你：我想实现用户登录功能，你觉得我需要考虑哪些问题？

AI：
1. 账号体系（邮箱 / 手机号 / 第三方登录 / 免密）
2. 密码策略（强度 / 加密传输 / 找回 / 过期）
3. Token 管理（存储 / 刷新 / 过期处理）
4. 安全性（防重放 / HTTPS / 验证码）
5. 用户体验（自动登录 / 记住密码 / 生物识别）
6. 合规（隐私政策 / GDPR / 未成年人保护）

你：好，按 1-5 实现，6 暂不做。第三方登录只要 Apple Sign-in。
```

**什么时候用**：**需求你自己都没完全想明白时**。让 AI 当产品经理帮你列清单，比你自己硬想快。

### 技巧 2：角色扮演（切换评估视角）

```text
你：你是我的 Tech Lead，审查这段代码并给出改进建议。

AI：[从 Tech Lead 视角审查，关注架构 / 可维护性]

你：现在你是产品经理。这个功能符合用户需求吗？

AI：[从产品视角评估，关注体验 / 指标]

你：现在你是 QA。这代码有哪些边界情况测试漏了？

AI：[列出 edge cases]
```

**什么时候用**：**对同一问题需要多个视角验证时**。相当于低成本的 review 委员会。
