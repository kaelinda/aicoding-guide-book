# 03-调试 Prompt

> 专门用来修 bug / 做性能优化的 Prompt 模板。**核心原则：先定位、再修改**。

---

## 为什么要单独一章讲调试 Prompt

Bug 修复和新功能开发**目标完全不同**：

| 维度 | 新功能 | 修 Bug |
|------|------|------|
| 上游 | 需求 | 现象 |
| 第一步 | 设计方案 | 定位根因 |
| 最大风险 | 功能做歪 | 改了症状没改根因 / 引入新 bug |
| Prompt 重点 | "做什么" | "不要急着改，先分析" |

所以调试 Prompt **不能照抄新功能的套路**。

---

## 模板 1：Bug 修复

```markdown
## Bug 描述
[一句话现象，比如"登录按钮点两次才响应"]

## 复现步骤
1. 打开 [页面]
2. 点击 [按钮]
3. 观察 [现象]

## 期望行为
[应该怎样]

## 实际行为
[实际怎样]

## 错误信息
​```
[完整的 stack trace / 崩溃日志 / console 输出]
​```

## 相关代码
文件：[具体路径]
方法：[方法名 / 行号]

## 环境
- iOS 版本：
- 设备型号：
- App 版本：
- 复现率：100% / 偶发 / 特定条件下

## 任务
先不要改代码：
1. 判断问题类型（状态 / 并发 / 生命周期 / 网络 / 依赖）
2. 列出你需要读的文件（最多 5 个，按优先级）
3. 给 3 个可能的根因，按可能性排序
4. 每个根因给"最快能验证它是不是"的方法
```

### 使用建议

- **"任务"这一段最关键**：明确说"先不要改代码"。AI 默认倾向"我看到 bug 就修"，会猜测性修复
- **错误信息给完整的**，不要摘录。AI 比人会看 stack trace，完整日志藏着答案
- **复现率很重要**：100% 复现是状态问题；偶发是并发 / 竞争条件 —— 两种根因方向完全不同

详见 [best-practices 反面案例 1](../../best-practices.md#反面案例-1不给上下文直接让-ai-改代码)。

## 模板 2：性能优化

```markdown
## 性能问题
[具体表现：卡顿 / 慢 / 高耗电 / 内存过大 / ...]

## 性能指标
- 当前：[fps / 响应时间 / 内存占用 / 启动耗时]
- 目标：[期望值]
- 测量工具：[Instruments Time Profiler / Hangs / Allocations]

## 场景
- [什么操作触发]
- [什么设备 / 什么数据量]

## 任务
先不要改代码：
1. 指出最可疑的 3 个性能问题点（文件 / 行号 / 理由）
2. 每点给一个**怎么用 Instruments 验证**的步骤
3. 最后给出修复优先级（哪个先做）

我会先跑 Instruments 确认，再让你出 patch。
```

### 使用建议

- **要求 Instruments 验证**：性能问题靠猜不靠谱，让 AI 给出可验证的诊断步骤
- **分"诊断 + 修复"两轮**：一轮猜根因 + 给验证步骤；真验证完再一轮修
- **给基准数据**：没有数字的性能优化是空谈（"优化一下"等于没说）

详见 [实战篇/03-Codex 实战/03-iOS 开发实战 § 场景 2](../../实战篇/03-Codex实战/03-iOS开发实战.md#场景-2性能优化诊断--优化) 的完整 demo。

## 链式 Prompt：迭代式修复

**场景**：bug 复杂，一次对话难以修到底。

```text
Round 1 — 诊断
你：[用 Bug 模板填好]
AI：给出 3 个根因假设 + 验证方法

Round 2 — 验证
你：已按你说的方法验证。根因是 A（不是 B/C）。
    具体证据：[贴日志 / 复现 / 测试结果]
    现在让我们修复 A。

AI：给 minimal patch

Round 3 — 验证修复
你：patch 应用后：
    - 原 bug 不再复现 ✅
    - 但是 [发现了新问题 / 测试失败]

AI：分析新问题，给下一步
```

**关键手势**：

- 每一轮**只做一件事**（诊断 → 验证 → 修复 → 再验证）
- **新的轮次要明确说"上一轮的结果是 X"**，别指望 AI 记住
- **修出新问题时**：先 `git reset --soft` 回退 patch，再让 AI 重新想；不要在错误的基础上继续叠

## 5 个常见错误与对照

### 错误 1：太简略

```text
❌ "写个页面"
✅ "创建一个商品详情页，包含图片轮播、标题、价格、购买按钮"
```

### 错误 2：太模糊

```text
❌ "优化一下"
✅ "优化列表滚动性能，目标 60fps，当前只有 30fps（iPhone 12 Pro，100 条数据）"
```

### 错误 3：缺上下文

```text
❌ "修复这个 Bug"
✅ "修复登录页邮箱验证 Bug：
    - 问题：邮箱带 + 号时验证失败
    - 文件：LoginViewModel.swift:67
    - 方法：validateEmail"
```

### 错误 4：期望过高

```text
❌ "开发一个像微信的 App"
✅ "开发聊天 App。第一步先实现消息列表页面。"
```

### 错误 5：不提供反馈

```text
❌ 生成代码后直接用，出问题再推翻重来
✅ 审查代码、提改进建议、迭代（Round 2、Round 3）
```

## 调试时避免的三条反模式

### 反模式 1：让 AI 猜根因并直接改

```text
❌ "登录按钮点两次才响应，帮我修"
```

AI 会猜一个根因（比如加防抖），掩盖真问题。见 [反面案例 1](../../best-practices.md#反面案例-1不给上下文直接让-ai-改代码)。

### 反模式 2：改完不验证就进下一步

```text
❌ AI："已修复，应该没问题了"
   你：好，继续做下一个功能

✅ 你：验证命令是什么？我跑完你再说"已修复"。
```

见 [反面案例 3](../../best-practices.md#反面案例-3相信应该能跑不验证)。

### 反模式 3：bug 越修越多还不停下

一个 bug 修出两个新的，**停下来回滚 + 重新想根因**，不要在错误基础上继续叠。

判断信号：

- 改了 3 次还没稳定修好
- 每次修完都引入 1+ 个新 bug
- 开始说"应该是 X 导致的，让我试试..."

这时 `git reset` 到 bug 修复前的状态，换个思路。
