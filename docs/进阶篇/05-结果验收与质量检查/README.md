# 05 结果验收与质量检查

> 确保 AI 生成的代码符合预期

---

## 为什么重要

AI 生成的代码可能：
- ✅ 功能正确
- ⚠️ 但有隐藏 bug
- ⚠️ 性能不好
- ⚠️ 安全风险

**验收检查必不可少！**

---

## 四层检查法

```
┌─────────────────────────────────────┐
│  第1层：功能验收                     │
│  代码是否实现了需求？                │
├─────────────────────────────────────┤
│  第2层：代码质量                     │
│  可读性、可维护性如何？              │
├─────────────────────────────────────┤
│  第3层：安全检查                     │
│  是否有安全漏洞？                   │
├─────────────────────────────────────┤
│  第4层：性能检查                     │
│  运行效率如何？                      │
└─────────────────────────────────────┘
```

---

## 第1层：功能验收

### 检查清单

- [ ] 核心功能是否实现？
- [ ] 边界情况是否处理？
- [ ] 错误情况有提示吗？

### 快速验证

```bash
你：帮我验证这个功能
- 输入：XX
- 预期输出：YY

Claude Code 会运行测试验证
```

---

## 第2层：代码质量

### 检查清单

| 项目 | 检查点 |
|------|--------|
| 命名 | 变量/函数名是否清晰 |
| 注释 | 关键逻辑有注释吗 |
| 结构 | 代码是否过于嵌套 |
| 重复 | 是否有重复代码 |

### 快速检查

```bash
你：检查代码质量问题，列出前3个最严重的
```

---

## 第3层：安全检查

### 检查清单

| 风险类型 | 检查项 |
|----------|--------|
| 注入 | 用户输入是否过滤 |
| 敏感信息 | 是否有 Key/Token 硬编码 |
| 权限 | 权限申请是否最小化 |

### 安全问题

```bash
你：检查这个代码的安全风险，列出所有问题
```

**常见高风险问题：**
- ❌ `eval(userInput)` 用户代码执行
- ❌ `password = "hardcoded"` 硬编码密码
- ❌ `SQL = "SELECT * FROM users WHERE id = " + userId` SQL 注入

---

## 第4层：性能检查

### 检查清单

| 场景 | 检查项 |
|------|--------|
| 循环 | 是否有嵌套循环 |
| 数据库 | 是否有 N+1 查询 |
| 内存 | 是否有大对象泄漏 |

### 性能测试

```bash
你：分析这段代码的性能问题
```

---

## 验收流程

```
需求 → AI 生成 → 第1层验证 → 第2层检查 → 第3层安全 → 第4层性能 → 通过 ✓
                         ↓
                    不通过 → 反馈 AI → 重新生成
```

---

## 常见问题

**Q: 检查太费时间？**
A: 可以让 AI 自动检查，节省时间

**Q: AI 检查不准确？**
A: 重要项目建议人工复核

**Q: 检查出问题怎么办？**
A: 把错误信息反馈给 AI，要求修复

---

## 小结

```
✅ 功能验收 → 需求是否实现
✅ 代码质量 → 可读性可维护性
✅ 安全检查 → 漏洞和风险
✅ 性能检查 → 运行效率
```
