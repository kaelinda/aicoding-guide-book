# 05-结果验收与质量检查

> AI 写完代码后，如何判断它真的可用

## 🎯 为什么这一章很重要

AI 最容易给人带来的错觉是：

- 代码很多
- 看起来很完整
- 解释也说得通
- 页面甚至能跑起来

于是人就会自然地产生一种判断：**“应该差不多好了。”**

但真实开发里，真正重要的不是“AI 有没有产出”，而是：

- 产出是否符合需求
- 逻辑是否正确
- 边界情况是否稳定
- 结构是否可持续维护

这一章的目标，就是帮你建立一套最基础但非常实用的验收方法。

## 🤔 为什么不能只看“能不能跑”

很多初学者在 AI 编程里会把“能运行”误认为“已经完成”。

但这两个判断差得很远。

一个功能即使能运行，也仍然可能存在这些问题：

- 逻辑不符合需求
- 边界条件没处理
- 错误提示缺失
- 状态流转混乱
- 代码结构难以继续修改
- 测试缺失或无效

所以，真正的工程习惯不是“生成完就收工”，而是“生成后进入验收流程”。

## ✅ 四层验收法

建议你把 AI 生成结果的检查，至少分成下面四层。

### 第一层：编译是否通过
这是最基础的一层。

检查内容：

- 有没有语法错误
- 有没有明显警告
- 工程是否能成功运行
- 新增文件是否都被正确引用

这一层解决的是“代码是否合法”。

但记住：**合法不等于正确。**

### 第二层：功能是否符合需求
这层要回答的问题是：

- 做出来的功能，是不是你要的那个功能？
- 有没有少做、漏做、错做？

检查方法：

- 把你最初的需求逐条拿出来对照
- 看 AI 有没有自己偷偷补了你没要的功能
- 看它有没有忽略你明确提到的要求

这是最容易被忽略的一层。

### 第三层：边界情况是否安全
很多 bug 都不是在“正常流程”里暴露的，而是在边界条件里。

常见边界包括：

- 空输入
- 重复点击
- 网络失败
- 数据为空
- 权限受限
- 极端数值
- 列表只有一条数据
- 删除最后一项

如果这些场景没验证过，就不要轻易认为功能“已经完成”。

### 第四层：结构是否可维护
这一层是最接近工程质量的部分。

检查内容：

- 命名是否清晰
- 是否有明显重复代码
- 状态流是否易懂
- 逻辑是否过度堆在 View 里
- 依赖是否过多
- 后续加功能时是否容易扩展

AI 有时会优先追求“快速给出一个能用结果”，但这不代表结构已经合理。

## 📋 一份通用验收清单

你可以把下面这份清单直接当成通用模板使用。

### 功能层

- [ ] 功能与需求一致
- [ ] 没有遗漏关键交互
- [ ] 没有明显做多或做偏

### 稳定性层

- [ ] 编译通过
- [ ] 运行正常
- [ ] 重复操作不会出错
- [ ] 空数据 / 空输入可处理
- [ ] 错误状态可见且可恢复

### 结构层

- [ ] 命名清晰
- [ ] 逻辑没有明显堆叠混乱
- [ ] 状态管理可理解
- [ ] 没有明显重复代码
- [ ] 没有不必要的复杂设计

### 协作层

- [ ] 你自己能解释主要实现思路
- [ ] 你知道下一步应该怎么改
- [ ] 你不是只能依赖 AI 才能继续维护

最后这一层尤其重要。

如果一段代码只能“交给 AI 看懂”，那它对你来说并不算真正可用。

## 📱 iOS 项目专项检查

因为这个仓库主要面向初级 iOS 开发者，所以建议你在通用验收之外，再加一层 iOS 专项检查。

### SwiftUI 页面检查

- [ ] 页面布局正常
- [ ] 不同状态下 UI 没有错位
- [ ] 按钮点击有反馈
- [ ] 文本和状态展示一致
- [ ] 导航行为符合预期

### 状态管理检查

- [ ] `@State` / `@Observable` 使用清晰
- [ ] 数据来源明确
- [ ] 页面刷新逻辑可解释
- [ ] 没有“看似刷新，实际上不同步”的问题

### 数据与持久化检查

- [ ] 数据创建正常
- [ ] 删除的是正确对象
- [ ] 重新进入页面后数据状态符合预期
- [ ] 本地持久化真正生效

### 体验层检查

- [ ] 空状态有提示
- [ ] 错误状态有提示
- [ ] 加载状态可感知
- [ ] 关键操作不会让用户困惑

## 💬 推荐让 AI 自检的 Prompt

当你已经拿到一份实现后，可以让 AI 帮你做“第二轮自检”。

例如：

```text
请不要重写代码，先帮我从下面 4 个角度检查当前实现：
1. 是否符合最初需求
2. 是否存在边界条件漏洞
3. 状态流是否清晰
4. 结构上有哪些可以改进但不需要大改的地方

请先输出检查报告，再给我最小修改建议。
```

这类问法，比直接说“帮我优化一下”有效得多。

## ⚠️ 最常见的验收误区

### 误区 1：只要能跑就算完成
这是最常见也最危险的误区。

### 误区 2：只看 happy path
只验证“理想流程”几乎一定会漏 bug。

### 误区 3：AI 自己说没问题，就真的没问题
AI 的解释能力很强，但解释能力不等于事实正确。

### 误区 4：为了省事跳过复查
如果你跳过验收，后面往往会用更多时间修更大的问题。

## 💡 建议养成的验收节奏

推荐你以后每次让 AI 产出代码后，都按这个顺序走：

1. 先编译和运行
2. 再对照需求逐条检查
3. 再测边界情况
4. 最后看结构是否能继续维护

哪怕每一步只花几分钟，整体质量都会明显提高。

## ✅ 小结

AI 编程真正的能力，不只是“让 AI 生成代码”，而是：

- 你知道该如何验收
- 你知道什么才算完成
- 你知道什么时候该继续追问，什么时候该自己改

记住这句话：

> **生成只是开始，验收才决定结果是否真正可用。**
