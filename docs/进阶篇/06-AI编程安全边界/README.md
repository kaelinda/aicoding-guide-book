# 06 AI 编程安全边界

> 知道什么能做，什么不能做

---

## 为什么要了解安全边界

AI 编程很强大，但**不是万能的**。了解边界才能：
- ✅ 安全使用 AI
- ✅ 避免踩坑
- ✅ 提高效率

---

## 安全边界分级

```
🟢 绿线区：安全，可以做
🟡 黄线区：注意，有风险
🟠 橙线区：谨慎，必须防护
🔴 红线区：禁止，非常危险
```

---

## 🟢 绿线区（安全）

### 可以放心做

- ✅ 写业务代码
- ✅ 重构现有代码
- ✅ 写单元测试
- ✅ 生成文档
- ✅ 代码审查
- ✅ 学习新框架

### 示例

```swift
// ✅ 安全：写业务代码
func calculateDiscount(price: Double) -> Double {
    return price * 0.1
}
```

---

## 🟡 黄线区（注意）

### 需要仔细检查

- ⚠️ 第三方库代码（可能不了解业务背景）
- ⚠️ 边界情况处理（可能漏掉 case）
- ⚠️ 复杂算法（可能有逻辑错误）

### 风险点

```swift
// ⚠️ 注意：边界情况
func divide(a: Int, b: Int) -> Int {
    return a / b  // ⚠️ b=0 会崩溃！
}
```

### 正确做法

```swift
// ✅ 检查后使用
func divide(a: Int, b: Int) -> Int? {
    guard b != 0 else { return nil }
    return a / b
}
```

---

## 🟠 橙线区（谨慎）

### 必须有防护措施

| 场景 | 风险 | 防护措施 |
|------|------|----------|
| 文件操作 | 误删/覆盖 | 先备份 |
| 数据库操作 | 数据丢失 | 先查询确认 |
| 网络请求 | 信息泄露 | 检查 URL |
| 系统命令 | 权限过大 | 限制命令范围 |

### 文件操作

```bash
# ⚠️ 危险命令
rm -rf ./src  # 可能删错目录

# ✅ 安全做法
# 先问 AI：这个操作会影响哪些文件？
# 确认后再执行
```

### 数据库操作

```swift
// ⚠️ 危险：直接删除
DELETE FROM users WHERE id = $id

// ✅ 安全做法
# 先查询确认
SELECT * FROM users WHERE id = $id
# 确认后再删除
DELETE FROM users WHERE id = $id
```

---

## 🔴 红线区（禁止）

### 绝对不能做

- ❌ 让 AI 操作生产环境数据库
- ❌ 让 AI 直接执行危险命令（`rm -rf /`）
- ❌ 让 AI 处理敏感信息（密码、密钥）
- ❌ 让 AI 执行无法撤回的操作

### 真实案例

```bash
# ❌ 绝对禁止：生产环境操作
$ claude "帮我清理一下生产数据库的日志"

# ✅ 正确做法
# 1. 先在测试环境测试
# 2. 人工审核 SQL
# 3. 备份数据
# 4. 准备回滚方案
# 5. 低峰期执行
```

### 敏感信息

```swift
// ❌ 禁止：硬编码密钥
let apiKey = "sk-xxxx-xxxxxxxx"

# ✅ 正确做法
# 1. 使用环境变量
# 2. 使用密钥管理服务
# 3. 配置文件隔离
```

---

## 安全使用 checklist

```
发送指令前检查：
□ 不是生产环境操作
□ 不是危险命令（rm -rf, drop table）
□ 不包含敏感信息
□ 有备份/回滚方案
□ 结果可验证
```

---

## 出问题怎么办

### 立即停止

```bash
# 按 Ctrl+C 停止 AI
# 不要让 AI 继续执行
```

### 评估影响

- 数据是否丢失？
- 系统是否受影响？
- 需要通知谁？

### 回滚方案

```bash
# Git 回滚
git revert HEAD
git reset --hard HEAD~1

# 数据库回滚
# 使用备份恢复
```

---

## 总结

```
🟢 绿线区：直接让 AI 做
🟡 黄线区：AI 做 + 仔细检查
🟠 橙线区：AI 建议 + 人工确认 + 备份
🔴 红线区：绝对禁止
```

**记住：AI 是工具，安全的最终责任在人类。**
