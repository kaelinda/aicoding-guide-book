# 06-AI 编程安全边界

> 让 AI 高效工作，不等于把一切权限都交给它

## 🎯 为什么要讲安全边界

AI 编程工具的一个巨大优势是：

- 它们能读代码
- 能改文件
- 能生成命令
- 有些还能直接执行操作

这让开发效率提升很快。

但也正因为这样，**一旦边界不清楚，风险也会一起放大。**

最常见的问题不是“AI 恶意做坏事”，而是：

- 它按自己的理解做了过度修改
- 它执行了你本来不想执行的命令
- 它动到了你不希望它碰的文件
- 它引入了额外依赖或风险配置
- 它在你没充分确认前做了不可逆操作

所以，安全边界不是“保守”，而是专业。

## 🤔 什么叫安全边界

简单来说，安全边界就是你要提前定义清楚：

- **AI 可以做什么**
- **AI 不能做什么**
- **哪些操作必须先问你**
- **哪些文件或目录不应该让它直接修改**

只要这些边界明确，AI 工具就会更像一个可靠助手；否则，它更像一个不够熟悉你项目习惯的临时实习生。

## 🔒 建议优先限制的 4 类操作

### 1. 删除类操作
这是最需要谨慎的一类。

例如：

- 删除文件
- 批量移动文件
- 清空目录
- 重置改动

这类操作往往具有不可逆性。

建议原则：

- 默认不要让 AI 自动执行删除类操作
- 如果必须执行，先要求它列出将要删除的内容
- 对关键目录一律人工确认

### 2. Git 高风险操作
AI 可能会建议使用一些“看起来能快速解决问题”的 Git 命令，但这些命令不一定适合你的仓库状态。

例如：

- `git reset --hard`
- `git clean -fd`
- `git push --force`
- 大范围 rebase / squash

建议原则：

- 不要默认允许 AI 自动执行高风险 Git 操作
- 所有会覆盖历史或改写提交的操作，都应该先人工确认

### 3. 依赖与环境修改
AI 很喜欢在“缺东西”的时候顺手建议：

- 安装一个新包
- 添加一个新 SDK
- 升级某个版本
- 改构建配置

这些操作短期看很方便，但长期可能带来：

- 依赖污染
- 版本冲突
- 构建不稳定
- 团队环境不一致

建议原则：

- 新增依赖必须说明原因
- 先给出备选方案
- 不要把“加依赖”当成默认解法

### 4. 敏感配置与密钥文件
这是必须明确保护的一层。

例如：

- `.env`
- API Key
- 证书与签名文件
- 发布脚本
- CI/CD 凭据配置

建议原则：

- 不要把密钥直接贴给 AI
- 不要让 AI 默认改动敏感配置
- 涉及凭据时，优先使用占位符示例

## 📁 建议重点保护的文件与目录

你可以把下面这些路径看成“默认敏感区”：

- `.env`
- `Secrets/`
- `fastlane/`
- `.github/workflows/`
- 证书、描述文件、签名配置
- 发布脚本
- 支付、登录、远程配置相关文件

不是说 AI 永远不能碰这些地方，而是：

> **一旦碰这些地方，必须切换到更谨慎的模式。**

## 🛠 AI 编程工具的安全使用建议

### Cursor

适合：
- 编辑器内逐步修改
- 边看边改
- 小范围协作

建议做法：
- 优先让它在局部文件中工作
- 大改前先让它输出计划
- 不要默认接受大面积 diff

### Claude Code

适合：
- 终端工作流
- 批量修改
- 生成和重构任务

建议做法：
- 对执行命令保持敏感
- 先看它准备做什么
- 高风险命令一律人工确认

### Codex CLI

适合：
- OpenAI 生态协作
- 复杂推理任务
- 命令行开发流程

建议做法：
- 明确审批模式
- 对联网、安装、执行命令设置预期
- 涉及系统级修改时不要盲目一路通过

## ✅ 一份最小安全约束清单

无论你用哪种工具，建议至少建立下面这些基本规则：

- [ ] 不自动执行删除类操作
- [ ] 不自动执行高风险 Git 命令
- [ ] 新增依赖前先说明原因
- [ ] 敏感文件改动前必须人工确认
- [ ] 涉及密钥时只用占位符示例
- [ ] 大范围重构前先输出修改计划
- [ ] 先局部验证，再扩大改动范围

这 7 条就已经能帮你避开很多常见问题。

## 💬 推荐约束 Prompt

你可以在任务开始前就把规则讲清楚。

例如：

```text
接下来请按下面的约束协助我：
- 不要删除文件
- 不要执行高风险 Git 命令
- 不要修改 .env、CI、发布脚本等敏感文件
- 如果需要安装依赖，请先说明原因
- 如果需要大范围修改，请先给出计划，等我确认后再继续
- 优先做最小可验证修改
```

这类约束非常有价值，因为它能显著减少“AI 自己发挥过头”的情况。

## ⚠️ 常见风险案例

### 案例 1：为了修一个 bug，顺手重构了整个模块
结果是：
- bug 可能修了
- 但 diff 变大很多
- review 成本上升
- 新问题也更难定位

### 案例 2：为了通过编译，自动加了依赖
结果是：
- 问题暂时解决
- 但项目复杂度上升
- 团队环境同步成本增加

### 案例 3：为了“清理环境”，建议 reset 或 clean
结果是：
- 问题可能没解决
- 本地未提交改动反而丢了

### 案例 4：把敏感配置直接贴给 AI
结果是：
- 风险暴露增加
- 还容易让后续示例代码里混入真实信息

## 💡 建议养成的安全习惯

### 1. 先局部，再全局
先让 AI 处理一个文件、一个函数、一个页面。

验证没问题，再扩大范围。

### 2. 先计划，再执行
对于重构、批量修改、目录调整这类任务，先让 AI 输出计划。

### 3. 先确认风险，再接受改动
不是所有“看起来专业”的建议都适合当前项目。

### 4. 把 AI 当成高效助手，不是最终决策者
真正负责项目结果的人，仍然是你。

## ✅ 小结

AI 编程的成熟用法，不只是“让它做更多事”，而是：

- 让它在合适的边界内做事
- 把高风险部分留给人工确认
- 在提效和可控之间找到平衡

记住这句话：

> **效率来自授权，但可靠性来自边界。**
