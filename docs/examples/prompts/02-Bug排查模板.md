# 02-Bug 排查模板

> 不要一出问题就让 AI 重写，先把 bug 讲清楚

## 🎯 这个模板适合什么时候用

当你遇到下面这些情况时，建议优先使用这个模板：

- 代码能编译，但行为不对
- 页面能显示，但交互异常
- 某个功能之前正常，现在不正常
- 你已经知道“哪里不对”，但不知道“为什么不对”
- 你希望 AI 先定位问题，而不是直接重写一版

## 🤔 为什么 Bug 排查要单独用模板

很多人遇到 bug 时第一反应是：

```text
帮我修一下
```

这种问法太宽泛，AI 很容易：

- 误判问题根源
- 一次改太多
- 把原来没问题的部分也改坏
- 为了“修好”而做过度重构

更好的方式是：

1. 先描述现象
2. 再交代预期行为
3. 再提供最关键的上下文
4. 最后要求它先分析，再给最小修改方案

## 🧩 通用模板

```text
我遇到一个 bug，请先不要重写代码，先帮我定位问题。

项目背景：
- 技术栈：{例如 SwiftUI + SwiftData}
- 当前文件/模块：{例如 TodoListView / WeatherViewModel}
- 最近做过的改动：{例如新增了搜索功能}

当前现象：
- {例如点击删除后删错对象}
- {例如输入关键词后列表不更新}
- {例如网络失败时一直卡在 loading}

我的预期行为：
- {例如删除当前选中的那一项}
- {例如搜索结果应立即刷新}
- {例如请求失败后显示错误提示并退出 loading}

请先帮我做这 4 件事：
1. 分析最可能的原因
2. 指出应该重点检查的代码位置
3. 给出最小修改方案
4. 修改后说明为什么这样改更稳妥

要求：
- 不要直接重写整个模块
- 优先最小修改
- 如果问题信息不足，请直接指出还缺什么
```

## ✅ 推荐补充的信息

如果你希望 AI 更快定位问题，建议补这些内容：

- 报错信息
- 触发步骤
- 哪一版开始出现问题
- 哪些行为是“稳定复现”的
- 你怀疑的文件或函数

这些信息越具体，定位越准。

## 💬 适合继续追问的方向

### 追问 1：让它列优先排查项

```text
请按优先级列出最可能的 3 个原因。
```

### 追问 2：让它只分析，不改代码

```text
先不要给修复代码，只分析问题根因。
```

### 追问 3：让它给最小 diff 思路

```text
请只给最小修改方案，不要顺手做结构重构。
```

### 追问 4：让它补验证步骤

```text
修复后我应该如何验证这个 bug 真的解决了？请给检查清单。
```

## 📱 iOS 示例

### 示例：搜索后删除错对象

```text
我遇到一个 bug，请先不要重写代码，先帮我定位问题。

项目背景：
- 技术栈：SwiftUI + SwiftData
- 当前模块：TodoListView
- 最近改动：新增了搜索和排序

当前现象：
- 搜索后列表会过滤
- 但删除某一项时，经常删错对象
- 清空搜索后原始列表和界面不同步

我的预期行为：
- 删除时应该删掉当前点击的正确对象
- 删除后界面和原始数据保持一致

请先帮我做这 4 件事：
1. 分析最可能的原因
2. 指出应该重点检查的代码位置
3. 给出最小修改方案
4. 修改后说明为什么这样改更稳妥

要求：
- 不要重写整个模块
- 优先最小修改
```

## ⚠️ 使用这个模板时的注意事项

### 注意 1：不要只说“有 bug”
一定要说明：
- 现象是什么
- 预期是什么
- 差异在哪里

### 注意 2：不要省略复现步骤
如果 bug 需要特定步骤才能复现，必须说出来。

### 注意 3：不要一开始就要求“大改”
定位阶段越克制，修复质量通常越高。

## ✅ 小结

Bug 排查的关键不是“让 AI 更快给答案”，而是让它先理解：

- 问题是什么
- 问题不是什么
- 你想修到什么程度
- 你不希望它顺手做什么

记住一句话：

> **描述 bug 的质量，往往决定修 bug 的质量。**
