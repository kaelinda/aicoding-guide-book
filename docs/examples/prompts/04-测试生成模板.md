# 04-测试生成模板

> 不要只让 AI 生成测试代码，要让它生成“有价值的测试”

## 🎯 这个模板适合什么时候用

适用于这些情况：

- 你已经有一段可运行代码，想补单元测试
- 你想让 AI 帮你补关键场景测试
- 你担心 AI 只会生成“形式上有测试、实际上没覆盖”的内容
- 你希望先确定测试范围，再让 AI 动手写

## 🤔 为什么测试生成也需要模板

AI 确实很会生成测试代码，但它也很容易犯下面这些问题：

- 只测 happy path
- 没测边界情况
- 断言太宽松
- 测试名字看起来很多，但覆盖没有价值
- 生成的测试和真实业务风险点对不上

所以，测试模板的核心目标不是“生成更多测试”，而是“生成更有意义的测试”。

## 🧩 通用模板

```text
我想为现有代码补测试，请先不要直接生成大量测试代码，先帮我确定测试范围。

项目背景：
- 技术栈：{例如 Swift + XCTest}
- 当前模块：{例如 TodoListViewModel / NetworkManager}
- 当前主要功能：
  - {例如 添加待办事项}
  - {例如 删除待办事项}
  - {例如 搜索过滤}

请先帮我做这 4 件事：
1. 列出最值得优先测试的功能点
2. 区分 happy path、边界情况、错误情况
3. 给出推荐测试清单
4. 再根据清单生成测试代码

要求：
- 优先测试关键业务逻辑
- 不要只生成表面测试
- 如果某些部分不适合单元测试，请指出原因
- 测试命名尽量清晰
```

## ✅ 推荐先测试什么

一般来说，优先级可以这样排：

### 第一优先级：核心业务逻辑
例如：
- 新增
- 删除
- 更新状态
- 搜索过滤
- 排序逻辑

### 第二优先级：边界情况
例如：
- 空输入
- 重复操作
- 极端数值
- 空列表
- 只有一条数据

### 第三优先级：错误处理
例如：
- 请求失败
- 数据解析失败
- 非法输入
- 权限不足

## 💬 适合继续追问的方向

### 追问 1：要求按优先级给测试计划

```text
请按 P0 / P1 / P2 给测试点排序。
```

### 追问 2：让它补边界情况

```text
请重点补边界情况，不要只测正常流程。
```

### 追问 3：让它解释为什么这样测

```text
请在生成测试代码前，先说明每个测试用例覆盖了什么风险。
```

### 追问 4：让它避免无价值测试

```text
请不要生成只验证 getter/setter 的低价值测试。
```

## 📱 iOS 示例

### 示例：为 TodoListViewModel 补测试

```text
我想为现有代码补测试，请先不要直接生成大量测试代码，先帮我确定测试范围。

项目背景：
- 技术栈：Swift + XCTest
- 当前模块：TodoListViewModel
- 当前主要功能：
  - 添加待办事项
  - 删除待办事项
  - 切换完成状态
  - 搜索过滤

请先帮我做这 4 件事：
1. 列出最值得优先测试的功能点
2. 区分 happy path、边界情况、错误情况
3. 给出推荐测试清单
4. 再根据清单生成测试代码

要求：
- 优先测试关键业务逻辑
- 不要只生成表面测试
- 测试命名尽量清晰
```

## ⚠️ 使用这个模板时的注意事项

### 注意 1：不要直接要求“生成完整测试”
先确认测试重点，结果通常会更好。

### 注意 2：优先测逻辑，不要先测表面 UI
尤其在初期，业务逻辑测试通常比 UI 测试更稳定、更高价值。

### 注意 3：测试生成后仍然要人工复查
AI 生成的测试也可能：
- 断言不够严谨
- 覆盖点不准确
- 测试名与行为不一致

## ✅ 小结

测试模板的意义，不是帮你“凑覆盖率”，而是帮你：

- 更快找到最该测试的点
- 更稳地覆盖关键逻辑
- 少写形式化、低价值测试

记住这句话：

> **测试的价值不在数量，而在它是否覆盖真正的风险。**
