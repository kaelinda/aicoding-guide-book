# iOS 调试与性能优化

> 面向 iOS 开发者的调试排障与性能优化速查，覆盖崩溃、网络、卡顿、内存、启动时间与线上问题定位。

---

## 为什么这页重要

很多 iOS 项目并不是“功能写不出来”，而是：

- 页面偶发白屏
- 列表滑动掉帧
- 图片导致内存暴涨
- 首屏启动慢
- 线上崩溃难定位
- 状态错乱但本地复现不稳定

AI 在这类问题上的价值，不是替你拍脑袋，而是帮你更快形成排查路径。

---

## 1. 调试总原则

建议按这条路径处理问题：

1. **先稳定复现**
2. **再缩小范围**
3. **再验证假设**
4. **最后补测试或监控**

不要一上来就大改代码。大多数线上问题都需要先确认：
- 是否必现
- 是否和网络有关
- 是否和线程有关
- 是否和生命周期有关
- 是否和数据边界条件有关

---

## 2. 常见问题分类

### 崩溃类
- 强制解包导致崩溃
- 数组越界
- EXC_BAD_ACCESS
- 非主线程更新 UI

### 性能类
- 启动慢
- 列表滑动掉帧
- 页面切换卡顿
- 图片内存占用高

### 网络类
- 请求偶发失败
- 登录态丢失
- Header / Cookie / Token 不一致
- API 返回结构变化导致解码失败

### 状态类
- SwiftUI 状态不同步
- 页面重复请求
- 返回上一级后状态丢失
- ViewModel 生命周期异常

## 3. 崩溃排查流程

### 第一步：拿到可靠信息
至少收集：
- 崩溃堆栈
- 触发路径
- 用户操作步骤
- 设备系统版本
- 最近发布改动

### 第二步：判断崩溃类型
高频判断方向：
- `fatal error`：大概率可选值、断言、越界
- `EXC_BAD_ACCESS`：大概率内存访问问题
- `Main Thread Checker`：大概率线程问题
- `DecodingError`：大概率接口字段变化

### 第三步：缩小到代码片段
重点看：
- 强制解包
- 索引访问
- 闭包捕获
- Task 内部状态更新
- UI 更新线程

### AI Prompt 模板

```text
请分析下面的 iOS 崩溃日志：
- 判断最可能的崩溃原因
- 给出排查步骤
- 指出代码中最值得优先检查的位置
- 给出修复建议
- 给出如何补测试或日志以避免再次发生
```

## 4. 网络问题排查

### 推荐工具
- Charles
- Proxyman
- Xcode Network Instruments

### 先检查什么
- URL 是否正确
- Header / Token 是否缺失
- 超时时间是否合理
- 请求体编码是否正确
- 接口返回字段是否变更

### 解码错误常见原因
- 字段缺失
- 类型变化
- 后端返回 `null`
- key 命名不一致

### AI Prompt 模板

```text
请帮我分析这个 iOS 网络请求问题：
- 请求代码如下
- 实际返回如下
- 解码模型如下
请判断问题出在哪一层，并给出修复建议。
```

## 5. SwiftUI 状态错乱排查

### 高频原因
- `@StateObject` / `@ObservedObject` 用错
- 在 `body` 中触发副作用
- View 重建导致状态丢失
- 异步任务重复触发
- 导航返回后对象被重新创建

### 排查思路
- 明确谁拥有状态
- 确认 ViewModel 生命周期
- 确认任务触发时机
- 检查是否有重复请求

### AI Prompt 模板

```text
请检查下面 SwiftUI 代码中的状态管理问题：
- 判断这里应该使用 @StateObject / @ObservedObject / @Binding 中的哪一个
- 解释当前问题为什么会导致状态错乱或重复请求
- 给出修复后的代码
```

## 6. 启动优化

### 常见慢启动原因
- 首屏前做了过多同步初始化
- 主线程读取大文件
- 配置、埋点、SDK 全部在启动时初始化
- 首屏页面依赖多个串行网络请求

### 优化方向
- 延迟初始化非关键模块
- 将可异步的工作移出首屏路径
- 减少主线程 IO
- 首屏先展示最小可用 UI

### 检查问题时可以问 AI

```text
请从 iOS 启动优化角度审查下面的 App 启动流程：
- 哪些工作不应放在首屏前
- 哪些初始化可以延迟
- 哪些逻辑应放后台线程
- 给出一个更合理的启动阶段划分
```

## 7. 列表卡顿与渲染性能

### 常见原因
- `body` 中做复杂计算
- 子视图过重
- 图片解码和缩放开销大
- 列表项频繁刷新
- 状态粒度过粗导致整页重绘

### 优化方向
- 预处理数据
- 拆分子视图
- 减少不必要的状态变更
- 图片异步加载 + 缓存
- 避免在 `body` 做高成本逻辑

### Instruments 重点看什么
- Time Profiler
- Main Thread Hotspots
- Allocations

---

## 8. 内存问题排查

### 重点症状
- 页面退出后内存不降
- 图片页打开后内存暴涨
- 列表滚动越久越卡

### 高频原因
- 图片缓存失控
- 强引用循环
- ViewModel / closure 捕获不当
- 大对象长期驻留

### 检查方向
- 是否存在 retain cycle
- 对象是否应该释放却没释放
- 图片是否做了压缩、降采样
- 是否缓存了不该缓存的大对象

### AI Prompt 模板

```text
请从内存优化角度审查下面这段 iOS 代码：
- 是否存在强引用循环风险
- 是否存在大对象长期持有问题
- 图片加载是否可能导致内存峰值过高
- 给出具体优化建议
```

## 小结

iOS 调试与性能优化最怕“凭感觉改代码”。更好的方式是：

- 先收集证据
- 再分类问题
- 再用工具验证
- 最后让 AI 帮你补排查路径、重构建议和测试思路

把 AI 用在“帮助你更快定位和验证”上，收益最高。
