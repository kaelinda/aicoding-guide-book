## 案例 3：Bug 修复

### 场景
用户报告：列表滚动时偶现崩溃。

### 使用工具
Claude Code + Codex

### 步骤

**1. 收集信息（5分钟）**
```
错误日志：
Thread 1: EXC_BAD_ACCESS
Crashed in UITableViewDataSource.tableView(_:cellForRowAt:)

代码位置：
ProductListViewController.swift:156
```

**2. 定位问题（10分钟）**
```
你：分析这个崩溃，可能的原因

Claude：
[分析代码]
发现问题：
1. 数据源数组在多线程访问
2. 未使用 @MainActor 标记
3. 数据更新未在主线程

建议修复：
1. 使用 actor 保护数据
2. 添加 @MainActor
3. 确保主线程更新
```

**3. Codex 深度分析（5分钟）**
```
你：深度分析这个多线程问题

Codex：
[推理分析]
根本原因：
- 并发访问竞争条件
- 数据源在后台线程修改
- UI 在主线程读取

最佳修复方案：
方案 1：使用 MainActor（推荐）
方案 2：使用 DispatchQueue.main.sync
方案 3：使用 NSLock
```

**4. 实施修复（10分钟）**
```
你：使用方案 1 修复

Claude：
[修改代码]
- 添加 @MainActor
- 使用 async/await
- 确保主线程访问
```

**5. 验证修复（5分钟）**
```
你：编写测试验证修复

Claude：
[生成并发测试]
[模拟多线程访问]
[验证无崩溃]

运行测试：✅ 通过
压力测试：✅ 1000 次无崩溃
```

### 结果
- ⏱ 实际用时：35 分钟
- 🎯 问题定位：准确
- ✅ 修复验证：通过
- 📈 稳定性：提升

### 经验总结
```
✅ 成功要素：
- Claude 快速定位
- Codex 深度推理
- 编写测试验证

❌ 避免的坑：
- 不要凭猜测修复
- 不要忽略并发问题
- 不要跳过压力测试
```

## 案例 4：多 Agent 协作

### 场景
需要开发一个完整的电商 App，包含用户、商品、订单三个模块。

### 使用工具
OpenClaw + 多 Agent 协作

### 步骤

**1. 任务拆解（10分钟）**
```
PM Agent：
分析需求...
任务拆解：
- Agent 1：用户模块（注册、登录、个人中心）
- Agent 2：商品模块（列表、详情、搜索）
- Agent 3：订单模块（购物车、下单、支付）

优先级：P0（核心功能）
预计时间：并行开发 2 小时
```

**2. 并行开发（2小时）**
```
Agent 1（用户模块）：
[创建文件结构]
[实现注册登录]
[添加测试]
进度：100% ✅

Agent 2（商品模块）：
[创建商品列表]
[实现搜索功能]
[添加测试]
进度：100% ✅

Agent 3（订单模块）：
[实现购物车]
[添加下单流程]
[集成支付]
进度：90% 🔄
```

**3. 集成协调（20分钟）**
```
PM Agent：
检测到 Agent 3 进度滞后
原因：支付接口对接问题

协调：
- Agent 1 协助调试支付接口
- Agent 2 提供商品数据支持

Agent 1 + Agent 3：
[联合调试]
[解决问题]
进度：100% ✅
```

**4. 集成测试（15分钟）**
```
Test Agent：
[运行完整流程测试]
[验证模块间通信]
[压力测试]

结果：
✅ 所有测试通过
✅ 性能达标
✅ 无崩溃
```

### 结果
- ⏱ 实际用时：2 小时 45 分钟
- 👥 Agent 数量：4 个（PM + 3 开发 + 1 测试）
- 🎯 完成度：100%
- 📈 效率提升：3x（vs 串行开发）

### 经验总结
```
✅ 成功要素：
- 清晰的任务拆解
- 合理的 Agent 分工
- PM Agent 协调

❌ 避免的坑：
- 不要让 Agent 职责重叠
- 不要忽略集成测试
- 不要缺少协调机制
```
