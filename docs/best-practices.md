## 案例 1：快速原型开发

### 场景
需要在 2 小时内完成一个用户登录页面的原型。

### 使用工具

### 步骤

**1. 描述需求（2分钟）**
```
创建一个用户登录页面，包含：
- 邮箱输入框（带格式验证）
- 密码输入框（可切换显示/隐藏）
- 登录按钮
- 忘记密码链接
- 注册入口

使用 SwiftUI 实现
```

**2. AI 生成代码（5分钟）**
```

- LoginView.swift（主视图）
- LoginViewModel.swift（视图模型）

直接可用，无需修改
```

```

**4. 迭代优化（5分钟）**

```

### 结果

- 📝 代码质量：生产可用

## 案例 3：Bug 修复

### 场景
用户报告：列表滚动时偶现崩溃。

### 使用工具
Claude Code + Codex

### 步骤

**1. 收集信息（5分钟）**
```
错误日志：
Thread 1: EXC_BAD_ACCESS
Crashed in UITableViewDataSource.tableView(_:cellForRowAt:)

代码位置：
ProductListViewController.swift:156
```

**2. 定位问题（10分钟）**
```
你：分析这个崩溃，可能的原因

Claude：
[分析代码]
发现问题：
1. 数据源数组在多线程访问
2. 未使用 @MainActor 标记
3. 数据更新未在主线程

建议修复：
1. 使用 actor 保护数据
2. 添加 @MainActor
3. 确保主线程更新
```

**3. Codex 深度分析（5分钟）**
```
你：深度分析这个多线程问题

Codex：
[推理分析]
根本原因：
- 并发访问竞争条件
- 数据源在后台线程修改
- UI 在主线程读取

最佳修复方案：
方案 1：使用 MainActor（推荐）
方案 2：使用 DispatchQueue.main.sync
方案 3：使用 NSLock
```

**4. 实施修复（10分钟）**
```
你：使用方案 1 修复

Claude：
[修改代码]
- 添加 @MainActor
- 使用 async/await
- 确保主线程访问
```

**5. 验证修复（5分钟）**
```
你：编写测试验证修复

Claude：
[生成并发测试]
[模拟多线程访问]
[验证无崩溃]

运行测试：✅ 通过
压力测试：✅ 1000 次无崩溃
```

### 结果
- ⏱ 实际用时：35 分钟
- 🎯 问题定位：准确
- ✅ 修复验证：通过
- 📈 稳定性：提升

### 经验总结
```
✅ 成功要素：
- Claude 快速定位
- Codex 深度推理
- 编写测试验证

❌ 避免的坑：
- 不要凭猜测修复
- 不要忽略并发问题
- 不要跳过压力测试
```

## 🧨 反面案例对照

> 前面 5 个案例都是"正确做法示范"，但真实开发里吃亏最多的往往是**看起来合理、实际上错**的做法。下面 6 组对照，每组按"❌ 错 → 出了什么 → ✅ 对 → 结果差异"展开。都是真实踩过的。

### 反面案例 1：不给上下文，直接让 AI 改代码

#### ❌ 错误做法

```text
你：LoginView 有个 bug，登录按钮点两次才响应，帮我修一下。
```

AI 直接给了一份 patch：加防抖、加 `disabled` 状态、重写 action 闭包。

#### 💥 出了什么

- AI 猜测根因，给的方案是**"防止快速双击"** —— 但真实根因是 `@StateObject` 每次 View 重建都新建了一个 VM，第一次点击在老实例上
- 防抖掩盖了症状，后面又出现"正常单点也偶尔不响应"
- 最终花了 2 小时回滚 + 重新定位

#### ✅ 正确做法

```text
你：LoginView 登录按钮点两次才响应。

先不要改代码：
- 读 LoginView.swift 和 LoginViewModel.swift
- 判断问题类型（状态 / 生命周期 / 并发 / 网络）
- 列出 3 个可能的根因，按可能性排序
- 每个根因给一个"最快能验证它是不是"的方法
```

AI 先定位到 `@StateObject` 误用，给出的方案是**改成 `@EnvironmentObject` 共享实例或用 `@Observable`**。

#### 📊 结果差异

| 维度 | ❌ 盲改 | ✅ 先定位 |
|------|------|------|
| 总耗时 | 2 小时（含回滚） | 15 分钟 |
| 回归风险 | 高（防抖副作用） | 低（根因修复） |
| 能复用 | 不能 | 能，这套提问模板写进 `CLAUDE.md` |

### 反面案例 2：一次 prompt 要求太多功能

#### ❌ 错误做法

```text
你：实现一个待办事项 app，要求：
- 增删改查 todo
- 按日期分组
- 支持优先级（高中低）
- 支持 tag
- 本地持久化（SwiftData）
- iCloud 同步
- Widget 展示
- 通知提醒
- 深色模式适配
```

AI 一口气生成了 20 多个文件。

#### 💥 出了什么

- 编译失败：`CKContainer` 权限没配、`WidgetKit` entitlement 没加
- 测试写是写了，但 SwiftData 的 test 容器初始化有问题，一个测试都跑不通
- 看起来"活儿干完了"，实际一步都没跑通
- 撤回 / 修复的成本比从零写还高

#### ✅ 正确做法

拆成 **walking skeleton → 逐步加功能**：

```text
Round 1：只做最小闭环
- 内存数据源的 todo 增删改查
- SwiftUI 列表展示
- 能编译、能跑、能增能删即可
- 不要 SwiftData、不要 iCloud、不要 Widget
```

跑通后再：

```text
Round 2：把内存数据源换成 SwiftData，其他不变
Round 3：加优先级和 tag（单独一个 PR）
Round 4：加通知
Round 5：iCloud 同步（单独一个 PR，最容易炸）
Round 6：Widget（独立模块）
```

#### 📊 结果差异

| 维度 | ❌ 一把梭 | ✅ 分 6 轮 |
|------|------|------|
| 首轮能跑 | 否 | 是 |
| 每轮 diff | 2000+ 行 | 150-400 行 |
| Review 能力 | 基本放弃 | 每轮能 review |
| 出错可定位 | 难 | 容易，锁定到某一轮 |

### 反面案例 3：相信"应该能跑"不验证

#### ❌ 错误做法

AI 写完，说"已实现，逻辑上应该能跑"。你相信了，commit、push、交给 QA。

```text
AI：已完成用户认证模块，包含登录、登出、token 刷新。逻辑上应该能覆盖所有分支。
你：好的，我提交了。
```

#### 💥 出了什么

- QA 测登出：app crash
- 定位：`token` 被 force-unwrap，登出后再访问触发 `nil`
- 原因：AI 写的测试只覆盖 happy path，登出后访问的边界没测
- 结果：延期一天修 bug + 写补充测试

#### ✅ 正确做法

用 [FAQ § 4.1 的验证 prompt](FAQ.md#-ai-生成的代码编译不过) 强制落地：

```text
你：刚才的改动：
- 改了哪些文件：<列出>
现在请：
1. 给出具体验证命令（xcodebuild / swift test / 手工点击路径）
2. 给出 3 个边界测试用例（不是 happy path）
3. 说明有哪些场景"看起来没问题但可能会出问题"
不要声称"应该没问题"。
```

AI：

- 登出后立刻再点"个人中心" → 会访问已清空的 token
- 登录失败后重试 → 如果上一次的 request 还没 cancel，可能有竞争
- app 被杀后台回来 → keychain 读失败时的 fallback 没测

你按这三条都验证一遍再 ship。

#### 📊 结果差异

| 维度 | ❌ 信 AI 自述 | ✅ 强制列边界 |
|------|------|------|
| QA 发现 bug | 2 个 | 0 |
| 返工 | 1 天 | 多花 5 分钟列清单 |
| 长期效果 | 信任感破产 | 每次都少两个 QA 回合 |

### 反面案例 4：把 AI 输出当权威，不核实 API

#### ❌ 错误做法

```text
AI：在 iOS 17+，可以使用 `UIApplication.shared.isBackgroundRefreshEnabled(for:)` 精确查询单个 bundle 的后台刷新状态。
你：好的，那我就这么用。
```

写上去编译报错：`cannot find 'isBackgroundRefreshEnabled(for:)' in scope`。

#### 💥 出了什么

这个 API **不存在**。AI 基于对命名惯例的"合理推测"编了一个看起来很自然的签名。
定位花了 20 分钟（一开始以为是 SDK 没更新）。

#### ✅ 正确做法

```text
你：你提到的 UIApplication.shared.isBackgroundRefreshEnabled(for:)，
请做两件事：
1. 给出 Apple 官方文档链接（developer.apple.com 的具体页面）
2. 用 Context7 MCP（或 WebFetch）搜一下这个符号是否存在
如果找不到，明确告诉我你之前说错了，不要再编一个类似的。
```

AI：之前描述有误，实际 API 是 `UIApplication.shared.backgroundRefreshStatus`（只能查询 app 自己的状态，无法按 bundle 查）。

#### 📊 结果差异

| 维度 | ❌ 盲信 | ✅ 要求溯源 |
|------|------|------|
| 踩坑率 | 每周 1-2 次 | 几乎为 0 |
| 返工时间 | 每次 10-30 分钟 | 0 |
| 习惯化 | 难 | 把"要求溯源"写进 `CLAUDE.md` 常驻 |

### 反面案例 5：长 session 不压缩不清空

#### ❌ 错误做法

早上开一个 Claude Code 会话，先讨论架构，然后修几个 bug，然后做新功能，下午继续，晚上也在同一个 session 里。

#### 💥 出了什么

- 上下文里混着 3 小时前的架构讨论、1 小时前已经 revert 的 bug 修复、现在正在做的功能
- AI 偶尔会用已经被撤掉的方案继续"演进"
- 响应越来越慢、token 费越来越贵
- Session 到晚上已经把"不要动 auth 模块"这种早上说好的约束丢了

#### ✅ 正确做法

- **一个 session 一件事**：bug 修完就 `/clear`，下一件新开
- **长任务到中段** `/compact` 或主动 `/summarize then /clear` 再继续
- **约束写进 `CLAUDE.md`**，不要依赖"AI 应该记得"

```text
你：我们先达成一致：
- 本 session 只处理登录模块的 bug
- 完成后我会 /clear 开新 session 做下一件
- 绝不顺手改 auth 以外的任何代码
```

#### 📊 结果差异

| 维度 | ❌ 全天一个 session | ✅ 按任务切分 |
|------|------|------|
| 晚间准确度 | 掉到 60% | 稳定 90% |
| token 成本 | 1x | 降 40-60%（新 session 不重读全部上下文） |
| 回滚难度 | 高（改动混在一起） | 低（每个 session 一套 commit） |

### 反面案例 6：不写 CLAUDE.md，每次 prompt 重复约束

#### ❌ 错误做法

项目里没有 `CLAUDE.md`。每次开新 session：

```text
你：我是 iOS 开发者，这个项目用 SwiftUI + MVVM，不要加依赖，不要改 auth 模块，
测试写在 tests/ 目录，每步要跑 xcodebuild 验证...（又说了 10 行）
```

#### 💥 出了什么

- 重复打字 10 行，每天浪费 10 次 → 100 行重复输入
- 偶尔漏掉一条约束，那一轮就会翻车
- 团队里每个人的约束略有不同，AI 输出风格漂移
- 新人加入，得口头交代半天

#### ✅ 正确做法

把稳定约束写进项目根的 `CLAUDE.md`（入 git）：

```markdown
# <项目名> 工作约定

- 技术栈：SwiftUI + MVVM + SwiftData (iOS 17+)
- 测试：tests/ 目录，每步用 `xcodebuild test -scheme X` 验证
- 禁改：auth/ 目录（走单独评审流程）
- 命名：Swift API 设计指南，camelCase
- 依赖：原则上不新增；确有需要先在 PR 描述里讨论
```

然后每次 session 只说**当次任务**即可，约束 Claude 自动读。

#### 📊 结果差异

| 维度 | ❌ 每次 prompt 重复 | ✅ CLAUDE.md 固化 |
|------|------|------|
| 每次 session 开场 | 10+ 行铺垫 | 1-2 行直接说任务 |
| 约束漂移 | 经常 | 几乎无 |
| 团队一致性 | 低 | 高（入 git，review 过） |
| 新人上手 | 需要口传 | 读 CLAUDE.md |

### 反面案例汇总

| 类型 | 核心错误 | 修正手势 |
|------|------|------|
| 1 | 不给上下文直接让 AI 改 | 先分析再动手 |
| 2 | 一把梭要求太多 | walking skeleton + 分轮 |
| 3 | 信 AI 自述的"应该没问题" | 强制列边界测试 |
| 4 | 把输出当权威 | 要求溯源 + MCP 查官方 |
| 5 | 长 session 混多任务 | 一 session 一任务 |
| 6 | 约束靠嘴说 | 写进 CLAUDE.md |

这 6 条不是并列的，它们有优先级：**1 最高**（影响所有后续），**6 最基础**（每个项目都该做）。先做 1 + 6，其余在踩坑时补。
