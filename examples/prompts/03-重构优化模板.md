# 03-重构优化模板

> 当功能已经能跑，但结构不够好时，如何让 AI 帮你安全重构

## 🎯 这个模板适合什么时候用

适用于这些场景：

- 功能已经能跑，但代码越来越乱
- 某个 View 或 ViewModel 变得太长
- 重复逻辑开始增多
- 你想拆层、提取组件、优化命名
- 你希望“改善结构”，但不想影响现有功能

## 🤔 为什么重构尤其容易翻车

AI 很擅长提出看起来很漂亮的结构方案，但重构的真正难点在于：

- 旧功能不能坏
- 旧行为要保持一致
- 改动范围不能失控
- review 成本不能太高

所以，重构最怕的不是“改不动”，而是“顺手改太多”。

## 🧩 通用模板

```text
我想对现有代码做重构优化，但请注意：
- 当前功能已经能运行
- 我希望优先改善结构，而不是增加新功能
- 请避免大范围无必要改动

项目背景：
- 技术栈：{例如 SwiftUI + SwiftData}
- 当前模块：{例如 UserListView / NetworkManager}
- 当前主要问题：
  - {例如 View 太长}
  - {例如 逻辑和 UI 混在一起}
  - {例如 重复代码较多}

请先帮我做这 4 件事：
1. 总结当前代码的主要结构问题
2. 给出一个最小可行重构方案
3. 按步骤说明应该先改哪里，再改哪里
4. 说明如何验证重构后功能没有回归

要求：
- 先不要重写整个模块
- 优先小步重构
- 保持现有功能行为不变
- 如果有风险较高的改动，请明确标注
```

## ✅ 什么时候应该先停下来，不要急着重构

如果你连下面这些问题都还说不清，建议先别重构：

- 当前功能是否稳定
- 哪些问题是真正痛点
- 重构目标是什么
- 你最不想动的部分是什么

重构不是为了“更优雅”，而是为了：

- 更容易维护
- 更容易改功能
- 更容易定位问题

## 💬 适合继续追问的方向

### 追问 1：让它给分阶段计划

```text
请把重构拆成 3 个阶段，每个阶段都能独立验证。
```

### 追问 2：限制改动范围

```text
先只处理命名和结构拆分，不要改业务逻辑。
```

### 追问 3：要求保留行为一致

```text
请明确说明哪些行为必须保持完全一致。
```

### 追问 4：要求列出回归风险

```text
请列出这次重构最容易导致回归的 3 个点。
```

## 📱 iOS 示例

### 示例：把过长的 SwiftUI 页面拆分

```text
我想对现有代码做重构优化，但请注意：
- 当前页面已经能运行
- 我希望优先改善结构，不增加新功能
- 请避免大范围无必要改动

项目背景：
- 技术栈：SwiftUI
- 当前模块：TodoListView
- 当前主要问题：
  - View 太长
  - 搜索、列表、工具栏逻辑都堆在一起
  - 子视图和状态没有清晰拆分

请先帮我做这 4 件事：
1. 总结当前结构问题
2. 给出最小可行重构方案
3. 按步骤说明先拆哪些子视图
4. 说明重构后如何验证行为没变

要求：
- 先不要大改业务逻辑
- 优先小步拆分
- 保持现有页面行为一致
```

## ⚠️ 使用这个模板时的注意事项

### 注意 1：重构目标要具体
不要只说“帮我优化一下”。

你最好明确说：
- 是想拆组件
- 想整理状态
- 想提取服务层
- 还是想统一命名

### 注意 2：一定要强调“行为不变”
如果不强调，AI 可能会把重构顺手变成改需求。

### 注意 3：先小步，再大步
尤其是现有代码已经能工作时，小步重构比一步到位稳得多。

## ✅ 小结

重构模板的核心价值，是帮你把“想整理代码”这件事做得更可控。

记住这个原则：

> **先定义重构边界，再让 AI 动手。**
