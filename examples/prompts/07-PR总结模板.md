# 07-PR 总结模板

> 让 AI 帮你把改动讲清楚，而不是只罗列文件变化

## 🎯 这个模板适合什么时候用

适用于这些场景：

- 你准备提交 PR
- 你已经完成一批改动，想整理成清晰说明
- 你想把“做了什么、影响哪里、怎么验证”讲明白
- 你希望团队成员更容易 review

## 🤔 为什么 PR 总结值得单独做模板

因为很多 PR 描述会出现这些问题：

- 只写了“优化代码”
- 没说改动范围
- 没说风险点
- 没说怎么验证
- 看起来有描述，但 review 价值不高

好的 PR 总结，不只是说明“改了什么”，更要说明：

- 为什么改
- 改动影响什么
- 有哪些风险
- reviewer 应该重点看哪里

## 🧩 通用模板

```text
请基于当前改动，帮我生成一份清晰的 PR 总结。

项目背景：
- 技术栈：{例如 SwiftUI + SwiftData}
- 当前改动目标：{例如 增加搜索功能 / 重构列表结构 / 补测试}

请输出下面这些内容：
1. 本次改动摘要
2. 主要改动点（按模块归类）
3. 影响范围
4. 潜在风险或注意事项
5. 验证方式 / 测试方式
6. reviewer 建议重点关注的部分

要求：
- 语言清晰
- 不要空泛
- 不要只罗列文件名
- 优先从功能和风险角度总结
```

## ✅ 推荐输出结构

建议输出成这样：

- 背景
- 改动摘要
- 主要改动
- 风险说明
- 验证方式
- reviewer 关注点

这样 reviewer 能更快抓到重点。

## 💬 适合继续追问的方向

### 追问 1：要更短版本

```text
请再给我一版更适合 GitHub PR 描述区的精简版本。
```

### 追问 2：要更适合团队 review

```text
请突出 reviewer 最需要优先关注的风险点。
```

### 追问 3：要按用户影响来写

```text
请按“用户可感知变化 / 内部结构调整 / 风险点”三个部分重写。
```

## 📱 iOS 示例

### 示例：TodoListViewModel 搜索重构 PR

```text
请基于当前改动，帮我生成一份清晰的 PR 总结。

项目背景：
- 技术栈：SwiftUI + SwiftData
- 当前改动目标：重构 TodoListViewModel 的搜索与删除逻辑，避免搜索后删错对象

请输出下面这些内容：
1. 本次改动摘要
2. 主要改动点（按模块归类）
3. 影响范围
4. 潜在风险或注意事项
5. 验证方式 / 测试方式
6. reviewer 建议重点关注的部分

要求：
- 语言清晰
- 不要空泛
- 不要只罗列文件名
```

## ⚠️ 使用这个模板时的注意事项

### 注意 1：不要只写“优化”“重构”
这些词太空，需要具体说明优化了什么、重构了什么。

### 注意 2：不要省略风险说明
哪怕风险很小，也值得写明 reviewer 重点看哪里。

### 注意 3：不要把验证方式写得太笼统
比起“已测试通过”，更好的是写：
- 测了哪些场景
- 怎么验证
- 哪些边界情况已覆盖

## ✅ 小结

PR 总结模板的价值，不只是帮你写得更像样，而是让协作更顺畅。

记住这句话：

> **好的 PR 描述，能减少 reviewer 猜你的改动意图。**
