# 01-需求澄清模板

> 在开始写代码前，先把需求问清楚

## 🎯 这个模板适合什么时候用

当你遇到下面这些情况时，建议先不要让 AI 直接写代码，而是先用这个模板：

- 需求还很模糊
- 你脑子里有想法，但表达不够完整
- 你担心 AI 一上来就生成错误方向的代码
- 你希望先把边界条件、技术约束和实现顺序讲清楚

这是一个非常值得养成的习惯。

## 🤔 为什么要先做需求澄清

很多 AI 协作失败，并不是因为 AI 写代码能力不够，而是因为一开始的问题就没有定义清楚。

常见表现：

- 功能范围不清楚
- 边界条件没说
- 技术栈没说
- 优先级没说
- 验收标准没说

结果就是：AI 会补全很多“你没说但它自己猜的东西”。

所以，先澄清需求，通常比“直接开始写”更省时间。

## 🧩 通用模板

你可以直接复制下面这段，再按自己项目替换内容：

```text
请你先不要写代码，先帮我把需求澄清清楚。

我的项目背景：
- 技术栈：{例如 SwiftUI + SwiftData}
- 当前正在做的模块：{例如 Todo 列表页}
- 目标用户：{例如 iOS 初学者 / 内部工具用户}

我现在想实现的功能：
{用 1~3 句话描述你的目标}

请你先帮我补全下面这些内容：
1. 这个功能的核心目标是什么
2. 最小可行版本应该包含哪些功能
3. 还可能有哪些边界条件
4. 这个功能适合拆成哪几个步骤实现
5. 哪些部分最容易出错
6. 我应该如何验收这个功能是否完成

要求：
- 先不要写代码
- 先输出澄清后的需求文档
- 语言尽量清晰，适合初学者理解
- 如果需求里有歧义，请直接指出
```

## ✅ 模板输出应该长什么样

理想情况下，AI 应该先给你一份结构清晰的需求整理，而不是直接开始写实现。

建议它至少输出这些部分：

- 功能目标
- 最小版本范围
- 边界条件
- 风险点
- 实施步骤
- 验收标准

如果它跳过这些直接写代码，你可以把它拉回来。

## 💬 补充追问模板

如果 AI 的第一次澄清还不够细，可以继续追问：

### 追问 1：让它明确边界条件

```text
请继续补充这个需求的边界条件：
- 空输入
- 重复操作
- 网络失败
- 数据为空
- 权限不足
```

### 追问 2：让它缩小范围

```text
请把这个需求收敛成最小可行版本。
先不要考虑进阶功能。
```

### 追问 3：让它按步骤拆解

```text
请把这个需求拆成 3 到 5 个实现步骤，按先后顺序排列。
```

### 追问 4：让它补验收标准

```text
请为这个需求补一份验收清单，确保我能在写完后逐项检查。
```

## 📱 iOS 场景示例

下面是一个更贴近你这个仓库定位的例子。

### 示例：Todo 列表页需求澄清

```text
请你先不要写代码，先帮我把需求澄清清楚。

我的项目背景：
- 技术栈：SwiftUI + SwiftData
- 当前模块：待办事项列表页
- 目标用户：个人使用的 iPhone 用户

我现在想实现的功能：
创建一个待办事项列表页，支持新增、删除、标记完成、搜索。

请你帮我补全：
1. 最小可行版本要包含哪些功能
2. 搜索和删除同时存在时，数据流应该如何设计
3. 这个页面最容易出现哪些状态管理问题
4. 应该按什么顺序实现
5. 验收时应该检查哪些点

要求：
- 先不要写代码
- 先输出结构化需求
- 保持内容简洁、清晰
```

这个问题问完后，你通常会得到比“直接帮我做个 Todo 页”稳定得多的结果。

## ⚠️ 使用这个模板时的注意事项

### 注意 1：不要把所有想法一次塞进去
如果你一次给太多背景，反而容易失焦。

建议保留最关键的信息：
- 技术栈
- 当前模块
- 目标功能
- 限制条件

### 注意 2：先定最小版本，再谈扩展
很多项目一开始就把“高级功能”说进去了，导致 AI 一次做太多。

建议永远先问：

- 最小版本是什么？
- 先做哪一部分？

### 注意 3：让 AI 明确指出歧义
不要假设 AI 会自动提醒你问题不清楚。

你最好明确写：

```text
如果需求中存在歧义，请直接指出。
```

## ✅ 小结

需求澄清模板的核心价值，是帮你把“模糊想法”变成“可执行任务”。

记住这个顺序：

1. 先讲背景
2. 再讲目标
3. 再补约束
4. 再要求拆解
5. 最后要求验收标准

只要这一步做得好，后面的代码生成质量通常会明显提高。
