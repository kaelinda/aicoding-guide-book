# 05-Code Review 模板

> 不要只让 AI 说“代码不错”，要让它真正做一次有价值的审查

## 🎯 这个模板适合什么时候用

适用于这些情况：

- 你已经写完一个功能，想做一次提交前检查
- AI 帮你生成了代码，你想让它做第二轮审查
- 你担心代码“能跑但不稳”
- 你想让 AI 从结构、边界、可维护性角度给出建议
- 你准备发 PR，希望先做一轮自查

## 🤔 为什么 Code Review 也需要模板

如果你只是对 AI 说：

```text
帮我 review 一下这段代码
```

它很容易给你一类“看起来很专业、实际上不够深入”的回答：

- 命名不错
- 结构清晰
- 建议加点注释

这些建议不一定错，但往往不够有用。

真正有价值的 Code Review，应该更关注：

- 逻辑风险
- 边界情况
- 状态管理
- 可维护性
- 是否有过度设计或隐藏耦合

## 🧩 通用模板

```text
请帮我对这段代码做一次 Code Review。

项目背景：
- 技术栈：{例如 SwiftUI + SwiftData}
- 当前模块：{例如 TodoListView / WeatherViewModel}
- 这段代码的目标：{例如 支持搜索、删除、状态切换}

请从下面 5 个角度审查：
1. 是否符合当前需求
2. 是否存在逻辑风险或边界条件漏洞
3. 状态流或数据流是否清晰
4. 代码结构是否容易继续维护
5. 有没有可以优化但不需要大改的地方

输出格式请按下面结构：
- ✅ 做得好的地方
- ⚠️ 建议改进的地方
- ❌ 明显问题或风险
- 🔧 建议的最小修改项

要求：
- 不要泛泛而谈
- 优先指出高风险问题
- 不要直接重写整个模块
- 如果某些部分没有问题，也请明确说明为什么
```

## ✅ 什么时候这类 Review 最有价值

下面这些场景尤其适合：

### 1. AI 刚生成完一段代码后
这时最容易出现“看起来完整，但隐藏问题很多”的情况。

### 2. 你自己已经写完，但准备继续扩功能前
这时适合先看结构是否稳，再决定要不要继续加东西。

### 3. 准备提交 PR 前
哪怕只是一次轻量检查，也常常能提前发现明显问题。

## 💬 适合继续追问的方向

### 追问 1：要求它只关注高风险问题

```text
请只保留高优先级问题，不要给太多低价值建议。
```

### 追问 2：要求它给最小修改建议

```text
请不要大改结构，只给最小修改方案。
```

### 追问 3：要求它从初学者可维护性角度审查

```text
请特别关注这段代码是否适合初学者继续维护。
```

### 追问 4：要求它检查是否“过度工程化”

```text
请检查这段实现是否为了看起来更专业而过度复杂。
```

## 📱 iOS 示例

### 示例：审查 TodoListViewModel

```text
请帮我对这段代码做一次 Code Review。

项目背景：
- 技术栈：SwiftUI + SwiftData
- 当前模块：TodoListViewModel
- 这段代码的目标：支持新增、删除、搜索、状态切换

请从下面 5 个角度审查：
1. 是否符合当前需求
2. 是否存在逻辑风险或边界条件漏洞
3. 状态流或数据流是否清晰
4. 代码结构是否容易继续维护
5. 有没有可以优化但不需要大改的地方

输出格式请按下面结构：
- ✅ 做得好的地方
- ⚠️ 建议改进的地方
- ❌ 明显问题或风险
- 🔧 建议的最小修改项

要求：
- 不要泛泛而谈
- 优先指出高风险问题
- 不要直接重写整个模块
```

## ⚠️ 使用这个模板时的注意事项

### 注意 1：不要把“挑小毛病”当成 Review
真正有价值的 Review，重点是风险和维护性，不是格式洁癖。

### 注意 2：不要把所有建议都照单全收
AI 的 review 结果仍然需要你判断优先级。

### 注意 3：Review 最好结合需求看
脱离需求审代码，很容易给出“正确但不重要”的建议。

## ✅ 小结

Code Review 模板的价值，是让 AI 真正参与“质量把关”，而不只是礼貌性点评。

记住这句话：

> **好的 Review 不是建议越多越好，而是能优先指出最值得改的问题。**
